📖 The story

A flag is read on practically every request, so it must cost nothing to evaluate — yet a flip must reach thousands of instances almost instantly. Those two goals pull in opposite directions.

The resolution: each service caches all flags locally and evaluates in memory (zero latency), while a push channel keeps those caches fresh within seconds. Deterministic hashing keeps a user in a stable rollout bucket.

How to think about it — step by step

Realize it's read on every request

Code checks 'is this flag on?' constantly. Why: the check must cost basically nothing.

Keep evaluation local

Each service caches all flags in memory and decides locally — no network call per check. Why: a network hop per request would be far too slow.

Solve the freshness problem

Push changes to those caches (or poll with a short timer) so a flip reaches everyone in seconds. Why: the whole point of flags is changing behavior without a deploy.

Make rollouts deterministic

Bucket users by hashing their ID, so the same user always lands in the same % group. Why: users shouldn't flicker in and out of a feature.

Accept brief staleness

A change may take a second or two to reach every server. Why: for features that's totally fine, and it buys huge speed.

1 · Requirements

Functional: turn features on/off, run % rollouts and targeting (by user, region) without deploys. Non-functional: read on nearly every request — must be ultra-low-latency — with changes propagating in seconds.

3 · The hard part — fast, fresh propagation

  • Local evaluation: the SDK holds all flags in memory and evaluates targeting rules locally — no network call per check.
  • Propagation: push (SSE/websocket/pub-sub) for instant updates, or poll with a short TTL; either way tolerate brief staleness.
  • Targeting: deterministic bucketing (hash(userId) % 100) so a user stays in the same rollout cohort.

4 · Trade-offs

  • Push (instant, more infra) vs poll (simple, laggy, more load).
  • Local cache means eventual consistency on flag changes — acceptable for almost all features.

Key sub-systems

  • Flag store — rules, variants, targeting.
  • Management API/UI — create/edit flags and rollouts.
  • Client SDK — caches flags, evaluates locally.
  • Propagation channel — streaming push or short-TTL poll.
  • Analytics — exposure logging for experiments.

Bottlenecks & what to optimize next

  • A per-request network call would be the bottleneck — evaluate locally in memory.
  • Propagation lag is the trade — push updates to refresh caches within seconds.

Common follow-up questions

How does a flag change reach thousands of servers fast?

SDKs subscribe to a push channel (or poll with a short TTL); the change refreshes their in-memory cache in seconds.

How do you keep a user in a stable rollout group?

Bucket by hash(userId) % 100 so the same user always lands in the same cohort across requests and sessions.

Isn't local caching inconsistent?

Briefly, yes — flags are eventually consistent. That's acceptable for nearly all features and buys zero-latency evaluation.
