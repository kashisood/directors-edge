Design a Distributed Message Queue (Kafka)

A technical deep-dive into the partitioned, replicated, append-only log that underpins modern event systems: partitions and ordering, consumer groups and offsets, replication, and delivery semantics.

📖 The story

The surprise of a high-throughput queue is that its speed comes from doing the dumbest fast thing: appending to a log on disk sequentially, not any clever in-memory structure.

From there the story is partitions — the log is split so many producers and consumers work in parallel, with order preserved within each partition. Replication with leader election makes it durable; consumer groups and offsets make it replayable. The recurring trade-off you’ll narrate: more partitions buy parallelism but cost you global ordering.

How to think about it — step by step

Ask where the speed comes from

From appending to a log on disk sequentially — the dumbest fast thing. Why: it explains every design choice that follows.

Split topics into partitions

Each partition is an ordered log; producers append by key. Why: partitions give parallelism, and order is kept within each one.

Let consumers track position

Consumer groups read by offset; one consumer per partition. Why: this is how you scale consumption and replay.

Make it durable

Replicate partitions; elect a new leader if one dies. Why: no data loss when brokers fail.

Name the trade-off

More partitions = more parallelism but no global ordering. Why: pick the key to match the ordering you actually need.

1 · Requirements

Functional: durable publish/subscribe — producers append messages to topics, multiple consumer groups read independently, with ordering and the ability to replay history. Non-functional: very high throughput (millions of messages/sec), durability (no data loss on broker failure), horizontal scalability, and bounded latency. This is the backbone that decouples services (it’s how the feed’s fan-out and the analytics pipeline are built).

2 · Capacity

Target millions of messages/sec with multi-day retention. The trick to that throughput is doing the dumbest fast thing: sequential append-only writes to disk (not random I/O), the OS page cache, zero-copy transfer to consumers, and batching + compression.

3 · Core concepts

  • Topic → split into partitions; each partition is an ordered, append-only log.
  • Producers append; a message’s key decides its partition (so same-key messages keep order).
  • Brokers store partitions; each partition has a leader + replicas (replication factor) for durability.
  • Consumer groups: each partition is read by exactly one consumer in a group — that’s how you parallelize. Consumers track their offset.

4 · Architecture

📍 Real example — topic, partition, offset

A Kafka message’s address is three things: topic → partition → offset. Picture a topic orders split into 3 partitions:

topic "orders"
 partition 0:  [0][1][2][3][4]      consumer-group "billing" committed offset = 3  (next read → 3)
 partition 1:  [0][1][2]
 partition 2:  [0][1][2][3]

produce(key="user-42", value=...)  →  hash(key) % 3  →  partition 1
   → every event for user-42 lands in partition 1, so its order is preserved

The diagram shows the topic 'orders' split into three partitions, each an append-only log numbered by offset. The consumer group remembers the last offset it processed in each partition, so after a restart it resumes from there. A message's key — for example user-42 — is hashed to choose its partition, which keeps all of that user's events in order.

What this teaches: the offset is just the position in an append-only log; the consumer group remembers the last offset it committed, so after a crash it resumes from there (and you can rewind the offset to replay history). The message key picks the partition via a hash, which is how you keep all events for one user/order strictly ordered — ordering is guaranteed within a partition, not across the whole topic. More partitions = more parallel consumers, but no global order.

5 · The hard parts

  • Ordering vs parallelism: order is guaranteed only within a partition. More partitions = more parallelism but no global order — so choose the partition key to match the ordering you actually need (e.g. by userId).
  • Durability: writes are acknowledged once replicated to the in-sync replica set (ISR); if the leader dies, a follower is elected leader with no data loss.
  • Delivery semantics: at-least-once by default (a crash before offset commit causes redelivery). Exactly-once needs an idempotent producer + transactional writes — available but adds overhead.
  • Why it’s fast: append-only sequential disk I/O + page cache + zero-copy + batching beat any clever in-memory structure.

6 · Scaling & failure

  • Scale by adding partitions (parallelism) and brokers (capacity); a controller rebalances partition leadership.
  • Broker failure: replicas + leader election keep the partition available; consumers resume from their committed offset.
  • Consumer failure: the group rebalances the dead consumer’s partitions to others; uncommitted messages are redelivered (hence idempotent consumers).
  • Retention: keep messages for a configured window (e.g. 7 days) to allow replay and slow consumers.

7 · Trade-offs

  • Partitions: more = throughput + consumer parallelism, but lose global ordering and add rebalancing/metadata overhead.
  • At-least-once (simple, needs idempotent consumers) vs exactly-once (correct, costlier).
  • Retention length: longer replay window vs storage cost.
Say it like a director

“It’s a partitioned, replicated, append-only log. I’d pick the partition key to preserve the ordering we actually need, size partitions for consumer parallelism, replicate for durability with leader election, and default to at-least-once with idempotent consumers — reaching for exactly-once only where the business truly requires it.”

Key sub-systems

  • Producers — append by key.
  • Brokers — store partitioned logs.
  • Partitions — ordered, replicated.
  • Consumer groups — read by offset.
  • Controller — leadership/rebalancing.

Bottlenecks & what to optimize next

  • Throughput scales with partitions — but more partitions cost global ordering and add overhead.
  • Slow consumers lag — retention + offsets let them catch up or replay.

Common follow-up questions

Where is ordering guaranteed?

Only within a partition. Choose the partition key (e.g. userId) to match the ordering you actually need.

How is it so fast?

Append-only sequential disk writes + page cache + zero-copy + batching beat any clever in-memory structure.

At-least-once vs exactly-once?

Default at-least-once (retries on consumer crash) with idempotent consumers; exactly-once needs an idempotent producer + transactions, at extra cost.
