Design Video Streaming (YouTube / Netflix)

📖 The story

This looks like a storage problem and is really a delivery problem. The money and the difficulty are in moving bytes to billions of eyeballs, not in a clever database.

The story has two arcs. On upload, the video is chopped into chunks and transcoded in parallel into many resolutions — turning a long serial job into a fast one. On playback, a multi-tier CDN serves small segments from the edge nearest each viewer, and the player adapts quality to the network on the fly. Obsess over cache-hit ratio and storage tiering, because egress and storage are where the cost lives.

How to think about it — step by step

Reframe it as delivery, not storage

The cost and difficulty are moving bytes to billions of screens. Why: it points you straight at CDNs.

Process uploads in parallel

Chop each video into chunks and transcode them into many resolutions concurrently. Why: turns a long serial job into a fast one.

Stream adaptively

Serve small segments via a manifest (HLS/DASH); the player picks quality based on bandwidth. Why: smooth playback on any network.

Push to the edge

A multi-tier CDN serves segments from near each viewer. Why: global low latency and origin offload.

Watch the money

Maximize cache-hit ratio and tier cold video to cheap storage. Why: egress and storage dominate cost at this scale.

1 · Requirements

Functional: upload a video; transcode it to multiple resolutions; stream it to a global audience with adaptive quality; track view counts. (Search/recommendations are separate systems.) Non-functional: enormous storage, very high read/egress bandwidth, global low-latency playback, reliable resumable uploads. This is overwhelmingly read-heavy, and egress bandwidth — not compute — is the dominant cost.

3 · API

  • POST /v1/uploads → returns a resumable upload URL (direct-to-blob, presigned) so bytes never flow through app servers.
  • GET /v1/videos/{id}/manifest → an HLS/DASH manifest listing the available renditions & segment URLs (on the CDN).
  • The player then fetches small segments (2–10s each) directly from the CDN, switching renditions as bandwidth changes.

4 · Data model

  • Metadata DB: videoId, owner, title, status (uploaded/transcoding/ready), available renditions, manifest URL.
  • Object storage: raw upload, plus transcoded segments per resolution (e.g. 240p…4K), plus the manifest files.
  • View counts: written via an async analytics pipeline (eventually consistent — an exact live count isn’t worth strong consistency).

7 · The hard parts

  • Transcoding throughput: split each video into chunks and transcode chunks in parallel across many workers, then stitch — turns a long serial job into a fast parallel one.
  • Global delivery: a multi-tier CDN serves segments from the edge closest to each viewer; popular videos are pre-pushed to edges.
  • Adaptive streaming: HLS/DASH segment the video so the client can switch quality mid-playback without re-buffering the whole stream.
  • Cost control: egress dominates, so maximize CDN cache-hit ratio; tier cold videos to cheaper storage; transcode only the resolutions actually watched for long-tail content.
📍 Real example — CDN cache hit vs origin pull

A video is chopped into small segments (e.g. 4-second .ts/.m4s chunks). The CDN edge server near the viewer caches them, so only the first viewer in a region pays the slow trip to origin:

viewer in Mumbai requests  /video/abc/720p/segment_017.ts

  edge (Mumbai) cache MISS  → pull from origin (US) ~180 ms, store with TTL=24h, serve
  next Mumbai viewer, same segment  → cache HIT ~8 ms   (never touches origin)

cache-hit ratio 95%  →  only 5% of requests reach origin
  → origin bandwidth and cost drop ~20x; viewers get a nearby, fast copy

The diagram shows a video split into small segments. The first viewer in a region misses the edge cache, so a segment is pulled once from the distant origin and stored nearby; every later viewer in that region gets it from the edge in a few milliseconds. At a ninety-five percent cache-hit rate, only about one request in twenty ever reaches the origin.

What this teaches: static, immutable segments are perfect for edge caching — popular content is served from a server milliseconds away while the origin handles only the long tail of misses. This is also how adaptive bitrate works: the player requests 720p segments when bandwidth is good and silently switches to 360p chunks when it drops, because each quality is just a different set of cached segment URLs. Measure success by cache-hit ratio and rebuffer rate, not raw throughput.

8 · Scaling & failure

  • Storage: object storage scales horizontally; hot/cold tiering controls cost.
  • Uploads are resumable, so a dropped connection continues rather than restarts; transcoding jobs retry idempotently with a dead-letter queue.
  • Delivery: if an edge fails, requests route to another edge or origin; players gracefully degrade quality rather than stall.
  • View counts ride the analytics pipeline — eventually consistent, never on the playback hot path.

9 · Trade-offs

  • Pre-transcoding many renditions costs storage & compute up front but delivers smooth adaptive playback — worth it for popular content; lazy-transcode the long tail.
  • CDN cost vs latency — we spend on edge delivery because viewer experience and origin offload demand it.
  • Eventual consistency for view counts — correctness here isn’t worth latency.
Say it like a director

“This is a delivery problem, not a database problem. Bytes go straight to object storage, a parallel chunked transcoding pipeline produces adaptive renditions, and a multi-tier CDN does the heavy lifting at the edge. I’d obsess over cache-hit ratio and storage tiering, because egress and storage are where the money goes at this scale.”

Key sub-systems

  • Upload service — resumable, direct-to-blob.
  • Transcoding pipeline — chunked, parallel renditions.
  • Object storage — segments + manifests.
  • CDN — multi-tier edge delivery.
  • Metadata + view counts — DB + async analytics.

Bottlenecks & what to optimize next

  • Egress/storage dominate cost — maximize CDN cache-hit ratio and tier cold video.
  • Transcoding is compute-heavy — chunk and parallelize; lazy-transcode the long tail.

Common follow-up questions

Why transcode into many resolutions?

Adaptive streaming lets the player pick the best quality for the current network; pre-making renditions enables smooth playback without re-buffering.

Why is the CDN essential?

Egress to billions of viewers from origin would be impossibly expensive and slow; the CDN serves segments from the edge nearest each viewer.

How do you control cost?

Maximize cache-hit ratio, tier rarely-watched video to cheap storage, and only pre-transcode popular content.
