📖 The story

Search is two systems wearing one coat. Offline, you turn the messy web into a tidy inverted index — a map from each word to the documents containing it. Online, you answer queries against that index in milliseconds.

At billions of docs the index is sharded across many machines, so a query scatters to all shards, each returns its best local hits, and a coordinator gathers and ranks them. The magic and the difficulty both live in ranking — blending text relevance with quality signals.

How to think about it — step by step

See it as two systems

Build the index (offline) and serve queries (online). Why: they have totally different constraints.

Build an inverted index

Map each word to the list of documents containing it. Why: it's what makes 'find docs with this word' fast.

Shard and scatter-gather

Split the index across machines; a query hits all shards, each returns its best, then merge. Why: the index is too big for one machine.

Rank well

Blend text relevance (BM25) with quality and freshness signals. Why: relevance is the actual product.

Mind the tail

The slowest shard sets your latency, so replicate and hedge. Why: scatter-gather is only as fast as its slowest part.

1 · Requirements

Functional: given a query, return relevant documents ranked best-first, fast. Non-functional: index a huge corpus, very low query latency, high availability, and reasonable freshness. The two halves are building the index (offline, batch) and serving queries (online, latency-critical).

2 · Capacity

Billions of documents and huge query volume mean the inverted index can’t live on one machine — it’s sharded across many, and queries scatter-gather across them within tens of milliseconds.

3 · API

  • GET /search?q=...&page= → ranked results + snippets + paging cursor.

4 · Data model

  • Inverted index: term → postings list (the doc IDs containing that term, with positions/frequencies) — sharded by term or by document.
  • Document store (for snippets/metadata) and a ranking-signal store (PageRank-like scores, freshness, clicks).

6 · The hard parts

  • Ranking: relevance = text match (TF-IDF / BM25) blended with quality/authority signals and freshness; this is the secret sauce.
  • Scatter-gather latency: a query must hit all shards and return fast — tail latency on the slowest shard dominates, so you over-replicate and hedge requests.
  • Index freshness: rebuild/merge segments incrementally so new docs appear without rebuilding everything.

7 · Scaling & failure

  • Shard the index (more shards = more parallelism); replicate each shard for availability and read throughput.
  • Cache popular queries; degrade gracefully (return fewer/partial results) if a shard is slow rather than stalling.

8 · Trade-offs

  • Freshness vs cost (incremental indexing vs full rebuild).
  • More shards = parallelism but more scatter-gather + tail-latency overhead.
  • Precomputed ranking signals (fast, slightly stale) vs computing at query time.

Key sub-systems

  • Crawler/ingest — documents in.
  • Indexer — builds the inverted index.
  • Index shards — partitioned postings.
  • Query service — scatter/gather + rank.
  • Query cache — popular searches.

Bottlenecks & what to optimize next

  • Tail latency on the slowest shard dominates — replicate shards and hedge requests.
  • Index freshness vs cost — incremental segment merges instead of full rebuilds.

Common follow-up questions

What makes search fast?

An inverted index (term→documents) so you never scan documents; sharded across machines and queried in parallel.

How do you rank results?

Blend text relevance (TF-IDF/BM25) with quality/authority and freshness signals; advanced systems use learning-to-rank.

Why is tail latency a problem?

A query waits for all shards; the slowest one sets the response time, so you replicate and hedge (send duplicate requests).

🧠 Remember it

Search = inverted index + sharded scatter-gather + ranking.

  • Inverted index (term → postings), sharded across machines.
  • Query scatters to shards, gathers, ranks (BM25 + signals).
  • Replicate shards + cache hot queries; tail latency is the enemy.
