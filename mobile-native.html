Mobile Native

You won't ship the app yourself, but if your org has a mobile presence you must speak fluently about how it's built, tested, signed, and released. App stores add constraints web doesn't have — review gates, signing, slow rollback — and leaders who ignore them get burned.

📖 The story

Mobile breaks a comfort the web gave you: you can no longer fix things instantly. The story to internalize is that the user owns the device and the stores own distribution — so releases are slow, gated, and hard to reverse.

Everything follows from that one fact: you cut builds early, test on real devices via TestFlight and Play tracks, roll out in phases, and keep a kill-switch (feature flags) because you can’t redeploy on demand.

You don’t need to write Swift or Kotlin — but if your org ships an app, you must plan around these constraints or they’ll bite you at the worst moment.

Why a director must know this

Mobile breaks several assumptions web engineers take for granted. You can't hot-fix instantly — a broken release may sit in an app-store review queue for hours or days, and users on old versions linger for months. Distribution goes through Apple and Google with their own rules. Signing and certificates can block a release entirely. If you lead a team that ships an app, these realities shape your planning, your risk, and your release strategy — so you need at least a high-level command of them.

The one-line mental model

Web: you control the server, so you ship and roll back in minutes. Mobile: the user controls the device and the stores control distribution, so releases are slower, gated, and hard to reverse — design for that.

Native vs cross-platform

The first architectural decision for any mobile effort. Know the trade-offs well enough to guide it.

ApproachWhat it isBest when
Native (Swift/SwiftUI for iOS, Kotlin/Jetpack Compose for Android)Two separate codebases using each platform's own tools.Max performance, deep platform features, large teams that can staff both. Games, hardware-heavy or flagship apps.
React NativeOne JavaScript/TypeScript codebase rendering real native UI; web-skill reuse.Shared logic across platforms, web teams extending to mobile, fast iteration. (Used by many large apps.)
FlutterOne Dart codebase, draws its own UI; very consistent look.Pixel-consistent UI across platforms, strong performance, teams willing to adopt Dart.
PWA / web wrapperA web app installed to the home screen or wrapped in a shell.Simple content apps, lowest cost, no app-store dependency — but limited device access.
Say it like a director

"I'd choose based on the team and the product, not fashion. If we have web/TypeScript strength and want one codebase shipping fast, React Native (often via Expo) is a strong default. If we're performance- or platform-feature-critical and can staff two teams, native wins. I'd name the trade-off: cross-platform buys speed and shared code; native buys ultimate performance and immediate access to new OS features."

Expo & React Native

Expo is a framework and platform on top of React Native that removes most of the painful native tooling. It's become the default way many teams build and ship React Native apps, so a leader should recognize the pieces.

Managed workflow

Expo handles the native build setup so developers work mostly in JS/TS. You can still "eject" or add native modules when you need them — it's no longer all-or-nothing.

EAS Build & Submit

Expo Application Services builds your iOS/Android binaries in the cloud (no need for every dev to own a Mac for iOS) and submits them to the stores — a managed mobile CI/CD pipeline.

EAS Update (OTA)

Over-the-air updates push JavaScript/asset changes straight to users without an app-store review — great for quick fixes. Caveat: only JS-layer changes; native changes still need a store release.

Expo Go vs dev builds

Expo Go is a sandbox app for quickly previewing during development; custom dev builds are used once you add native modules. Useful to know when the team discusses their workflow.

OTA updates — use with discipline

Over-the-air updates (EAS Update, or CodePush historically) are powerful for fixing JS bugs fast, but app stores require that OTA updates don't materially change the app's purpose or bypass review. Treat them for fixes and tweaks, not for shipping whole features around the review process.

Distribution & beta testing

Getting the app to users — and to testers first — works very differently on each platform. The vocabulary here matters in interviews.

TestFlight (iOS)

Apple's beta distribution. Internal testers get builds fast; external testers (up to 10,000) require a lighter Beta App Review. The standard way to get an iOS build into testers' hands before public release.

Google Play testing tracks

Android offers internal, closed, and open testing tracks, plus staged production rollout. Internal testing is near-instant; tracks let you widen the audience gradually.

App Store review

Apple reviews every public release against its guidelines (often hours to a day, occasionally longer). Rejections happen — plan schedules with review time and possible rejection in mind.

Play Store review

Google also reviews submissions; historically faster than Apple but still a gate. Both stores can reject for policy, privacy, or technical issues.

Say it like a director

"I plan mobile releases assuming review is part of the critical path — we cut the build early, get it into TestFlight and Play's internal track for real-device validation, and never assume a same-day hotfix is possible. For genuine emergencies on the JS layer we have an OTA path, but the store release is the one we schedule around."

Mobile CI/CD & code signing

Mobile pipelines have one big extra concern web doesn't: cryptographic signing. Apps must be signed with the right identity or they won't install or publish.

  • iOS signing: certificates and provisioning profiles tie the app to your Apple Developer account and allowed devices. Expired or misconfigured signing is a classic release blocker.
  • Android signing: a keystore signs the app; Play App Signing lets Google manage the signing key. Losing your keystore (pre-Play-signing) can mean you can't update your own app — treat signing keys as critical secrets.
  • Fastlane: a popular open-source tool that automates building, signing, screenshots, and store submission for both platforms.
  • EAS / cloud build: Expo's managed alternative; builds and submits in the cloud, handling much of the signing complexity.
  • Secrets: signing keys, certificates, and store API keys are sensitive — store them in a secrets manager / CI secure store, never in the repo (see Security).

Release management for mobile

Phased / staged rollout

Release to a small percentage of users first (Play staged rollout; iOS phased release), watch crash and adoption metrics, then ramp. Your canary for mobile.

Versioning

A user-facing version (semver, e.g. 3.2.0) plus an ever-incrementing build number the stores require. Know the difference; the build number must go up every submission.

Forced & soft upgrades

Because old versions linger, apps often check a minimum supported version and prompt (or force) an update. Plan for users who never upgrade.

Rollback reality

You can't truly "roll back" a shipped app — you halt the rollout and ship a fixed forward version (or OTA the JS fix). Feature flags and remote config are how you disable a bad feature instantly.

Feature flags / remote config

Ship code dark and toggle it server-side — the main way to control risk when you can't redeploy on demand. Essential for mobile.

Crash monitoring

Tools like Crashlytics/Sentry track crash-free-user rate per version — a key release-health metric you should watch during a rollout.

The director's mobile release reflex

Cut early, test on real devices via TestFlight / Play tracks, roll out in phases watching crash-free rate, keep risky features behind flags, and assume the store release is slow and one-directional. That posture prevents most mobile release disasters.

Mobile-specific concerns to speak to

Offline & sync

Phones lose connectivity. Good apps work offline and sync later — local storage, queues, and conflict handling. Ask "what happens on a train with no signal?"

Push notifications

Delivered via Apple's APNs and Google's FCM. A re-engagement lever and an operational concern (delivery, opt-in, fatigue).

App size & performance

Download size affects install rates; cold-start time and UI smoothness (no "jank") affect retention. Battery and memory matter on constrained devices.

Privacy & permissions

Apple's App Tracking Transparency and privacy "nutrition labels", Google's Data Safety section, and runtime permissions. Mishandling privacy gets you rejected or fined.

On-device security

Secrets live in the iOS Keychain / Android Keystore, not plain storage. Consider certificate pinning and that the client is untrusted — never hide real secrets in the app binary.

Fragmentation

Especially on Android: many OS versions, screen sizes, and device capabilities. Define a supported-device matrix and test against it.

What's the difference between native and cross-platform mobile development?Core

Native means building separately for each platform with its own tools — Swift/SwiftUI on iOS, Kotlin/Compose on Android — for maximum performance and immediate access to new OS features, at the cost of two codebases. Cross-platform (React Native, Flutter) means one codebase that targets both, trading some performance and platform-immediacy for shared code, faster iteration, and a smaller team. As a leader I'd choose based on our team's skills and the product's demands, and I'd name that trade-off explicitly rather than picking a framework for its hype.

What is TestFlight and why does it matter?Core

TestFlight is Apple's beta-distribution service: it lets us get pre-release iOS builds onto real testers' devices — internal testers almost immediately, and up to 10,000 external testers after a light beta review — before a public App Store release. It matters because it's how we validate a build on real hardware and gather feedback safely. The Android equivalent is Google Play's internal/closed/open testing tracks. I treat getting a build into these as a standard step in every mobile release, not an afterthought.

How does releasing a mobile app differ from releasing a web service, and how does that change your planning?Deep

Three big differences. First, distribution is gated — Apple and Google review releases, so "ship now" can mean hours or days. Second, you can't force users to upgrade, so old versions live on for months and your backend must support them. Third, rollback isn't instant — you halt a staged rollout and ship a fix forward, or OTA a JS fix. So I plan with review time and possible rejection in the schedule, keep risky features behind feature flags / remote config, roll out in phases watching crash-free rate, and design APIs to be backward-compatible. The mobile release posture is fundamentally more cautious than web.

Say it like a director

"On mobile I assume the store is in my critical path and that I can't hotfix on demand — so we cut early, roll out gradually, and keep an instant kill-switch via flags rather than relying on redeploys."

What are over-the-air (OTA) updates, and what are their limits?Core

OTA updates (e.g. Expo's EAS Update, or CodePush historically) push JavaScript and asset changes directly to installed apps without going through app-store review — great for fast bug fixes. The key limit is that they only cover the JS/content layer; anything touching native code still requires a full store release. There's also a policy boundary: stores require that OTA updates don't materially change the app's purpose or be used to dodge review. So I'd use OTA for quick fixes and tweaks with discipline, and never to smuggle whole features past the review process.

Why is code signing such a big deal in mobile, and how do you manage the risk?Trap

Apps must be cryptographically signed or they won't install or publish — iOS uses certificates and provisioning profiles, Android uses a keystore. Expired certificates or a lost keystore are classic, painful release blockers; pre-Play-App-Signing, losing your keystore could mean you literally can't update your own app. I manage it by treating signing keys as critical secrets in a secure store, using Google's Play App Signing and managed services (EAS, or Fastlane match) so keys aren't scattered on laptops, and monitoring certificate expiry so it never surprises us mid-release.

Your org is starting a mobile app from scratch. How do you decide the tech and set it up to ship reliably?Deep

I'd start from the team and the product, not the framework. If we have strong web/TypeScript talent and want one codebase shipping quickly, React Native via Expo is a strong default — EAS gives us cloud builds and OTA without every dev owning a Mac. If we're performance- or platform-critical and can staff two teams, native. Then I'd invest early in the things that make mobile reliable: a CI/CD pipeline with managed signing, TestFlight/Play internal tracks wired in, staged rollouts, crash monitoring on crash-free-user rate, and feature flags so we can disable bad features without a release. I'd also set a supported-device and minimum-version policy up front. The goal is that shipping becomes routine and reversible despite the store constraints.

A critical bug is live in your mobile app. Walk me through your response.Trap

First, contain without waiting for a store release: if the feature is behind a flag or remote config, disable it instantly; if it's a JS-layer issue and we use OTA, push the fix that way. In parallel, halt any in-progress staged rollout so we stop widening the blast radius. I'd assess impact via crash-free rate and analytics by app version. Then ship the proper fixed build through review — flagging Apple/Google for expedited review if it's severe — while communicating clearly to stakeholders and, if needed, users. Afterward, a blameless post-mortem: why did it reach production, and what guardrail (better testing on real devices, a flag, a staged rollout) prevents the class of bug. The lesson I'd reinforce: on mobile, the kill-switch is the flag, not the deploy.

🧠 Remember it

On mobile, the store is in your critical path and you can’t hotfix — design for it.

  • Cut early, test on real devices, roll out in phases.
  • Feature flags are your kill-switch, not a redeploy.
  • Choose native vs cross-platform by team + product, not hype.
