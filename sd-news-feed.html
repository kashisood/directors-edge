Design a News Feed (Twitter / X timeline)

A full, technical walk-through of the canonical feed problem: how to serve a personalized home timeline to hundreds of millions of users with instant reads, using the precompute-on-write (fan-out) pattern.

📖 The story

The whole design hangs on one number: people read their feed far more than they post — roughly fifty to one. So you do the expensive work on the rare event (the post) to make the constant event (the read) trivial.

That’s ‘fan-out on write’: when you post, workers push the tweet into each follower’s precomputed timeline, so reading is just fetching one ready-made list. The twist in the tale is the celebrity — pushing one post to 100M followers is a write storm — so you special-case them with pull-on-read. That hybrid is the punchline.

How to think about it — step by step

Anchor on the read/write ratio

People scroll their feed far more than they post — roughly 50 to 1. Why: this one number decides the whole design.

Decide when to do the work

Do the heavy lifting on the rare write, not the constant read: when you post, push the tweet into each follower's ready-made timeline. Why: reading then becomes 'grab one prepared list' — instant.

Store the prepared feeds

Keep each user's timeline as a capped list in Redis; keep the tweets themselves in a scalable store. Why: memory makes reads microsecond-fast.

Find where the plan breaks

A celebrity with 100M followers would trigger 100M writes per post. Why: spotting this is the crux of the question.

Patch it with a hybrid

For huge accounts, don't push — pull their recent posts at read time and merge. Why: avoids the write storm while keeping normal users fast.

1 · Requirements

Functional: a user can post a tweet, follow/unfollow others, and load a home timeline (recent posts from people they follow), with new posts appearing within a second or two. Non-functional: read-heavy, highly available, low read latency (the home timeline must feel instant), and eventual consistency is acceptable (a post showing up a second late is fine).

2 · Capacity estimation

The dominant fact is the 50:1 read:write ratio. That single number drives the whole design: we pay extra work on the (rare) write to make the (constant) read cheap — this is the precompute-the-timeline idea.

3 · API

  • POST /v1/tweets → { text, mediaIds } returns tweetId (idempotency key supported).
  • GET /v1/feed?limit=50&cursor=<opaque> → a page of hydrated posts + next cursor (cursor pagination, never offset).
  • POST /v1/follows / DELETE /v1/follows/{userId}.

4 · Data model

  • Tweets (wide-column store, e.g. Cassandra): partition by tweetId (Snowflake ID, time-sortable). Columns: authorId, text, mediaIds, createdAt.
  • Social graph: follows(followerId, followeeId) — sharded; queried as “who follows X” (for fan-out) and “who does X follow”.
  • Timeline cache (Redis): per user, a capped list of recent tweetIds — e.g. timeline:{userId} => [id, id, …] (latest ~800). This is the precomputed home feed.

5 · Architecture — the write path (fan-out)

On a post, we (1) persist to the Tweet store synchronously, then (2) enqueue a fan-out job. Async workers look up the author’s followers and prepend the tweetId to each follower’s timeline list in Redis. Now a home-timeline read is just “read one Redis list” — O(1)-ish and microseconds fast.

6 · Architecture — the read path

📍 Real example — read replica vs shard

Two different database scaling moves get confused constantly. They solve different problems:

// READ REPLICAS — one primary takes writes, copies stream to read-only replicas
write  →  [ PRIMARY ]  —replicates→  [ replica 1 ]  [ replica 2 ]  [ replica 3 ]
reads  →                                  ↑ SELECT spread across replicas
   helps when: 95% of traffic is reads (like a feed). Hurts nothing but write capacity.
   gotcha: replication lag — a replica may be 50ms behind, so a user can post and not see it yet.

// SHARDING — split the data itself across independent primaries
user_id % 4 == 0  →  [ shard 0 ]   (its own primary + replicas)
user_id % 4 == 1  →  [ shard 1 ]
user_id % 4 == 2  →  [ shard 2 ]
user_id % 4 == 3  →  [ shard 3 ]
   helps when: writes or total data are too big for one machine.
   gotcha: cross-shard queries (e.g. "feed from 100 friends on 4 shards") get hard — you fan out and merge.

What this teaches: reach for read replicas first — they’re cheap and a feed is read-heavy, but they don’t add write capacity and they lag. Reach for sharding only when one primary can’t hold the writes or the data; it scales writes but makes anything spanning shards (joins, “merge my friends’ posts”) your problem to solve in application code. Most large systems use both: shard the data, then replicate each shard.

7 · The hard part — fan-out at scale

Pure push (fan-out on write) breaks for celebrities: pushing one post from an account with 100M followers means 100M Redis writes — a single tweet causes a write storm. The standard answer is a hybrid.

StrategyHowCost
Fan-out on write (push)Precompute timelines; reads are trivial.Write amplification; brutal for celebrities.
Fan-out on read (pull)At read time, merge recent posts from everyone you follow.Cheap writes; expensive, slow reads.
Hybrid (the answer)Push for normal accounts; for a handful of celebrity accounts, don’t push — pull their recent posts at read time and merge into the cached timeline.Balances both; a little merge complexity.
Say it like a director

“I’d precompute timelines with fan-out on write because reads dominate 50-to-1, but I’d special-case high-follower accounts to pull-on-read so one celebrity post doesn’t trigger a hundred-million-write storm. That hybrid is the crux of the design.”

8 · Scaling & bottlenecks

  • Timeline cache: shard Redis by userId (consistent hashing); cap each list (~800 ids) so memory is bounded; replicate for availability.
  • Hot keys: a viral thread can hammer one tweet’s row — add a read-through cache layer and possibly replicate hot tweets.
  • Tweet store: Cassandra scales writes horizontally; partition by tweetId; reads are batched key lookups during hydration.
  • Stateless services (Post, Timeline, Fan-out) scale out behind the load balancer; fan-out runs off a queue to absorb spikes.

9 · Failure & reliability

  • Cache loss: a timeline list is a cache, not source of truth — on miss we rebuild it by pulling recent posts from followees, so Redis going down degrades latency, not correctness.
  • Fan-out lag: the queue may back up under spikes; posts appear a little late — acceptable per our eventual-consistency requirement, and a dead-letter queue catches failures.
  • Targets: set an SLO on timeline read latency (e.g. p99 < 200ms) and an error budget; the Tweet store is replicated across AZs.

10 · Trade-offs summary

  • We trade extra write work (fan-out) for cheap reads — correct because reads dominate.
  • We accept eventual consistency (slightly delayed posts) to get availability and low latency.
  • We add hybrid complexity for celebrities to avoid catastrophic write amplification.

Key sub-systems

  • Post service — persists tweets, triggers fan-out.
  • Fan-out workers — push tweetIds into follower timelines.
  • Timeline cache — per-user Redis lists.
  • Tweet store — Cassandra, source of truth.
  • Graph service — follower lookups.

Bottlenecks & what to optimize next

  • Celebrity fan-out is the bottleneck — switch high-follower accounts to pull-on-read.
  • Timeline-cache memory grows — cap list length and evict inactive users.

Common follow-up questions

How do you handle a user with 100M followers?

Don't push their posts; pull-on-read — fetch their recent tweets at read time and merge into the (mostly precomputed) timeline.

What if the timeline cache is lost?

It's only a cache — rebuild a user's timeline by pulling recent posts from who they follow; correctness is unaffected, latency dips briefly.

Chronological vs ranked feed?

Chronological is simple (just the list); ranked needs a scoring service over candidate posts — more infra and a feedback loop.
