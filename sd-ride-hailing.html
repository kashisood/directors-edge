Design Ride-Hailing (Uber / Lyft)

A technical deep-dive into real-time matching: ingesting a million-plus moving locations per second, geospatial indexing (geohash / QuadTree), proximity matching, and splitting ephemeral location data from durable trip state.

📖 The story

The beating heart of this system is one question asked millions of times a second: who’s near this rider, right now? You can’t scan every driver, so you need an index where nearby points sit together.

That’s geospatial indexing — geohash or a QuadTree — held in memory and sharded by region. Drivers stream locations in (a million-plus writes a second), the matching service queries nearby cells and ranks by ETA, and a durable trip state machine takes over once matched. The elegant split: locations are ephemeral (lose them, drivers re-ping in seconds), but the trip is durable because that’s where money and safety live.

1 · Requirements

Functional: drivers continuously report location; a rider requests a ride and is matched to a nearby available driver; both track the trip in real time; ETA and (surge) pricing. Non-functional: very high write throughput (constant location pings), low-latency matching, geographic partitioning, high availability; trip state must be durable, live locations need not be.

3 · The key idea — geospatial indexing

You can’t scan every driver to find who’s nearby. You need an index where nearby points are stored together:

  • Geohash: encode (lat, long) into a short string; a shared prefix means physical proximity. “Find nearby” = query the rider’s cell + its 8 neighbors (to handle edges).
  • QuadTree / S2 cells: recursively subdivide space; cells get smaller where driver density is high — better for uneven density (dense city vs sparse suburb).

5 · The matching deep-dive

  • Find candidates: compute the rider’s cell, gather available drivers in it + neighboring cells.
  • Rank: by real ETA (road network/traffic), not straight-line distance.
  • Dispatch safely: “lock” the offered driver for a few seconds so they’re not double-offered; on decline/timeout, fall back to the next candidate.
  • Moving targets: drivers move constantly, so the index is updated ~every few seconds and stale entries expire — the data is intentionally ephemeral.

6 · Scaling & failure

  • Geo-shard the index by region/city so a city’s write load and queries stay local; route requests by location.
  • Boundaries: a rider near a shard edge needs neighbor cells that may live on another shard — query both, or overlap cells.
  • Hot cities: scale that region’s shards independently; surge pricing is computed per region from local supply/demand.
  • Durability split: live locations are in-memory and disposable (lose them, drivers re-ping in seconds); trip state is persisted and replicated, with idempotent dispatch so a retry never creates two trips.

7 · Trade-offs

  • Geohash (simple, fixed cells) vs QuadTree/S2 (adapts to density, more complex) — pick based on how uneven your density is.
  • In-memory speed for locations (no durability needed) vs durable, consistent trip state (money + safety).
  • Geo-sharding scales writes but complicates cross-boundary matching.
Say it like a director

“The crux is indexing a million-plus moving points per second so I can answer ‘who’s near this rider’ in milliseconds — that’s a geohash or QuadTree index, in memory, sharded by region. Locations are ephemeral so I optimize for write speed there, and keep the trip itself in a durable, idempotent state machine because that’s where correctness and money live.”

Key sub-systems

  • Location ingest — high-write driver pings.
  • Geo index — geohash/QuadTree, in memory, region-sharded.
  • Matching service — nearby + ETA ranking.
  • Trip service — durable state machine.
  • Pricing — per-region surge.

Bottlenecks & what to optimize next

  • 1M+ location writes/sec is the firehose — keep locations in memory, sharded by region, with TTL expiry.
  • Cross-boundary matching — query neighbor cells; hot cities scale independently.

Common follow-up questions

How do you find nearby drivers fast?

A geospatial index (geohash/QuadTree) so nearby drivers share a cell key; query the rider's cell plus neighbors.

Why keep locations in memory and not a durable DB?

They change every few seconds and don't need durability — if lost, drivers re-ping in seconds; this enables the huge write rate.

What must be durable?

The trip state machine (money, safety) — persisted and replicated, with idempotent dispatch so a retry never creates two trips.
