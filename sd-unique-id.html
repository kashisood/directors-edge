📖 The story

Every distributed system needs unique IDs, and the naive answers all fail at scale: a database counter is a bottleneck, and UUIDs are huge and unsortable.

Snowflake threads the needle: pack a timestamp, a machine ID, and a counter into 64 bits, generate locally with no coordination, and get IDs that even sort by time. The one gotcha to name is the clock moving backwards.

How to think about it — step by step

Write down what 'good' means

Unique, generated fast from many machines, ideally sortable by time, and small. Why: these requirements rule options in and out.

Reject the obvious-but-wrong answers

A database counter is a single bottleneck; a random UUID is huge and unsortable. Why: showing why simple answers fail proves you understand the constraints.

Combine three ingredients

Pack a timestamp + a machine ID + a per-millisecond counter into 64 bits (Snowflake). Why: the machine ID prevents collisions and the timestamp keeps IDs sortable — with zero coordination.

Spot the failure mode

If a machine's clock jumps backwards, you could repeat IDs. Why: good engineers name the edge case (clock skew) and handle it by pausing.

State the trade-off

Sortable IDs are great for database locality but leak roughly when something was created. Why: a small privacy cost worth knowing.

3 · The hard parts

  • No coordination: the machine-ID bits keep hosts from colliding; the sequence handles many IDs within the same millisecond.
  • Clock skew: if the clock moves backwards (NTP), you must pause or borrow from sequence — the main failure mode to handle.
  • Machine IDs: assigned via a registry (ZooKeeper/etcd) or config.

4 · Trade-offs

  • UUIDv4 (zero coordination, 128-bit, unsortable) vs DB auto-increment (sortable, single bottleneck) vs Snowflake (balanced, clock-dependent).
  • Time-sortable IDs help DB index locality but leak creation time.

Key sub-systems

  • ID library — runs in-process on each host (no service call).
  • Machine-ID registry — assigns the per-host bits (ZooKeeper/etcd/config).
  • Clock — NTP-synced; the timestamp source.
  • Sequence counter — per-millisecond, per host.

Bottlenecks & what to optimize next

  • A central allocator would be the bottleneck — generate locally instead.
  • Clock skew is the real risk — detect backwards jumps and pause/borrow from the sequence.

Common follow-up questions

What if the clock moves backwards?

Stop issuing IDs until the clock catches up, or borrow from the sequence bits; never risk a duplicate.

Why not just use UUIDs?

UUIDv4 needs no coordination but is 128-bit and not time-sortable, hurting DB index locality. Snowflake is compact and sortable.

How are machine IDs assigned?

From a coordination service (ZooKeeper/etcd) or static config; the bit budget caps how many hosts you can have.
