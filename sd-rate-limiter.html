Design a Distributed Rate Limiter

A technical deep-dive into rate limiting across a fleet: algorithm choice (token bucket), atomic distributed counters in Redis, the latency-vs-accuracy trade-off, and fail-open behavior.

📖 The story

The trap in a rate limiter is thinking it’s about counting. It’s really about counting the same thing from many machines at once, in under a millisecond, without lying.

So the design is a token bucket whose check-and-decrement is a single atomic operation in a shared Redis — enforced right at the gateway, before any expensive work. The senior touches are two: push a local bucket per node that reconciles periodically to kill the per-request hop, and fail open so the limiter can never become the outage it was meant to prevent.

How to think about it — step by step

Pin down the limit's scope

Per user? Per IP? Per API key? And is the limit global across all servers? Why: a 'global' limit is much harder than a per-server one.

Pick a counting model

A token bucket: a bucket refills at a steady rate, each request takes a token, empty means reject. Why: it allows short bursts but enforces an average — what most APIs actually want.

Share the count safely

Keep buckets in a shared store (Redis) and do check-and-decrement atomically. Why: many servers hit the same key at once, so the update must be a single safe operation.

Kill the latency cost

At extreme scale, keep a local bucket per server that syncs with Redis periodically. Why: avoids a network round-trip on every single request.

Decide the failure behavior

If the limiter's store is down, allow traffic (fail open). Why: the limiter must never be the thing that takes your whole API down.

1 · Requirements

Functional: cap requests per key (per user, IP, or API key) to N per time window; configurable rules per endpoint; reject over-limit requests with HTTP 429 + a Retry-After header. Non-functional: works across a fleet of gateway nodes (a global limit, not per-node), adds sub-millisecond latency, and is itself highly available.

2 · Capacity

If the fleet handles ~1M req/s, the limiter does ~1M checks/s. Each check is a tiny in-memory counter op — the design constraint is latency and coordination (all nodes must agree on the count), not storage.

3 · Algorithm choice

4 · Architecture

5 · The hard part — distributed accuracy without latency

  • Atomicity: many gateway nodes hit the same key concurrently, so check-then-decrement must be atomic — do it in one Redis Lua script (or INCR + TTL) to avoid races.
  • Centralized vs local: a shared Redis is accurate but adds a network hop on every request. A common optimization is a local token bucket per node that periodically reconciles with Redis — near-zero latency at the cost of slight over-admission.
  • Hot keys: a single abusive key concentrates load on one shard; mitigate with local pre-checks and key sharding.
📍 Real example — a circuit breaker

When a downstream service is failing, hammering it with retries makes things worse. A circuit breaker wraps the call in a tiny state machine that “trips” after too many failures and stops sending traffic for a cooldown:

  CLOSED  —— failures ≥ threshold (e.g. 5 in 10s) ——→  OPEN
  (normal,                                                  (fail fast: reject
   calls flow)                                               instantly, don’t call)
     ↑                                                         |
     |                                                  after cooldown (e.g. 30s)
  success                                                        |
     |                                                           v
  CLOSED  <—— 1 trial call succeeds ——  HALF-OPEN  —— trial fails ——→ OPEN
                                          (let ONE request through to test)

What this teaches: CLOSED = healthy, calls pass through. After enough failures it flips to OPEN and fails instantly — this protects the struggling service and keeps your own threads from piling up waiting on timeouts (the real cause of cascading outages). After a cooldown it goes HALF-OPEN, lets a single probe through, and either recovers (back to CLOSED) or re-trips. Libraries like Resilience4j and Envoy/Istio do exactly this; pair it with a timeout, a bounded retry with backoff, and a fallback.

6 · Failure mode — fail open or closed?

If the limiter’s store is unreachable, do we allow or block traffic? Most systems fail open (allow) so the rate limiter never takes the whole API down — better to briefly under-protect than to cause an outage. Security-critical limits may fail closed. Pair with a local in-memory fallback bucket so we still get coarse protection during a Redis blip.

7 · Trade-offs

  • Accuracy vs latency/cost: exact global counting needs a round-trip per request; local+sync trades a little accuracy for speed.
  • Fail open (availability) vs fail closed (protection) — choose per use case.
Say it like a director

“Token bucket in a shared Redis with an atomic check-and-decrement, enforced at the gateway. At very high volume I’d push a local bucket per node that reconciles periodically to kill the per-request hop, and I’d fail open so the limiter can never become the cause of an outage.”

Key sub-systems

  • Gateway middleware — enforces limits before real work.
  • Counter store — shared Redis holding token buckets.
  • Atomic update — Lua script for check-and-decrement.
  • Rules config — per-route/per-key limits.
  • Local fallback — in-memory bucket if Redis is unreachable.

Bottlenecks & what to optimize next

  • The shared counter is the hot spot — shard by key and use local buckets that sync periodically.
  • A single abusive key creates a hot shard — pre-check locally before the shared store.

Common follow-up questions

How do you count accurately across many servers?

Keep the bucket in a shared store and make check-and-decrement a single atomic operation (Redis Lua) so concurrent requests can't race.

How do you avoid a network hop on every request?

Keep a local token bucket per node that reconciles with the shared store periodically — near-zero latency, slight over-admission.

Fail open or closed if the store is down?

Usually fail open so the limiter never causes an outage; security-critical limits may fail closed.

🧠 Remember it

A limiter is distributed counting that must be atomic, fast, and fail-open.

  • Token bucket; atomic check-and-decrement in shared Redis.
  • Enforce at the gateway, before expensive work.
  • Fail open — never let the limiter cause an outage.
