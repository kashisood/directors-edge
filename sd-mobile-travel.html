📖 The story

‘Design a travel booking app’ bundles search at scale, payments, complex itineraries, and the most unforgiving UX in mobile: I’m at the gate and I need my boarding pass now.

The trick is to decompose it into flows and notice the one constraint that dominates — the boarding pass must work offline. That pushes you to an offline-first client, with the heavy lifting (search fan-out, booking) on the server.

How to think about it — step by step

Clarify scope first

Flights only, or hotels/cars too? An OTA that sells inventory, or metasearch that redirects? Logged-in trips? Why: it changes which subsystems you build.

Map the user journey

Search → results → detail → book → trip (itinerary, check-in, boarding pass) → post-trip. Why: mobile design is organized around screens/flows, not just servers.

Find the airport-critical screen

The boarding pass must work with no signal. Why: this one constraint (offline-first) drives the whole client design.

Make search feel fast

Fan out to many fare providers server-side, cache by (origin, destination, date), and stream results as they arrive. Why: flight search is genuinely slow; stream so the user sees something immediately.

Make booking safe

Idempotency keys so a retry never double-books; handle 3DS/SCA; support Apple/Google Pay. Why: mobile networks drop mid-payment, so retries are normal.

Design for offline + disruption

Cache itinerary + boarding pass locally and in the Wallet; push gate/time changes as Live Activities. Why: the value is highest exactly when connectivity is worst.

1 · Requirements & scope

Functional: search flights, compare/filter, book & pay, manage the trip (itinerary, check-in, boarding pass), and get disruption alerts. Non-functional: fast search despite slow providers, rock-solid booking, and an offline-first trip experience — the boarding pass must work in airplane mode.

2 · The core flows

  • Search — airports, dates, passengers.
  • Results — filter (stops, airline, time), sort (cheapest/best/fastest).
  • Detail — fare rules, baggage, seat selection.
  • Book — passenger info, payment, confirmation.
  • Trip — itinerary, check-in, boarding pass, notifications.

3 · Architecture

4 · Search — the expensive part

Flight search is one of the costliest queries in tech: a single route pulls dozens of fares from many providers of varying speed. Three techniques tame it:

  • Server-side fan-out — the backend queries providers in parallel and aggregates (the phone never does this).
  • Cache by (origin, destination, date) with a short TTL — fares go stale within minutes, so the cache is small-window but high-hit during a search session.
  • Stream results — show the first batch instantly and append more as slow providers respond, so the UI never feels frozen.

5 · Booking & payment

  • Idempotency keys — a dropped connection + retry must never double-book or double-charge.
  • 3DS / SCA — strong customer authentication (heavy in the EU) handled in-flow.
  • Apple Pay / Google Pay first-class — fewer form fields, higher conversion.
  • Optimistic UI — clear “processing your booking” state while the GDS confirms.

6 · Itinerary, boarding pass & offline

  • Offline-first — itinerary and boarding pass are cached on-device so they open in airplane mode or a dead-zone jet bridge.
  • Wallet integration — PKPASS (Apple Wallet) / Google Wallet; the barcode shows on the lock screen without unlocking.
  • Live updates — gate/seat/time changes pushed in real time; a Live Activity shows time-to-boarding for active flights.
  • Timezone-aware — depart in one zone, arrive in another; always show the right local times.

Key sub-systems

  • Search service — GDS fan-out, fare cache, result streaming.
  • Booking service — idempotent orders, payment/3DS, PNR creation.
  • Itinerary service — trips, segments, live status.
  • Mobile client — offline cache, Wallet passes, Live Activities.
  • Notification service — check-in reminders, disruption alerts.

Bottlenecks & what to optimize next

  • Flight search latency is the bottleneck — fan out, cache, and stream results.
  • Connectivity at the airport is worst when stakes are highest — make trip data fully offline.

Common follow-up questions

Why is flight search so slow?

Fares come from many providers (GDS) of varying speed, and fare rules are genuinely complex. You fan out in parallel, cache by route+date, and stream results so the user isn't blocked.

How do you stop a double-booking on a flaky network?

An idempotency key on the booking request — a retried request returns the original booking instead of creating a second.

How does the boarding pass work with no signal?

It's cached on-device and added to Apple/Google Wallet (PKPASS), so the barcode renders offline, even on the lock screen; updates apply when connectivity returns.

What separates a staff-level answer here?

Raising the offline-first constraint, booking idempotency, timezone-aware itinerary display, and the disruption-handling flow — not just search and itinerary CRUD.

🧠 Remember it

Mobile travel = server-side search fan-out + idempotent booking + offline-first trips.

  • Search: fan out to providers, cache by route+date, stream results.
  • Booking: idempotency keys, 3DS, Wallet pay.
  • Trip: offline boarding pass (Wallet), live disruption updates, timezone-aware.
