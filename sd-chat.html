Design a Chat System (WhatsApp / Messenger)

A technical deep-dive into real-time messaging at planet scale: persistent connections, routing a message to the recipient's live server, ordering, offline delivery, and end-to-end encryption.

📖 The story

Throughput isn’t the villain here — state is. Billions of messages is manageable; the hard part is holding hundreds of millions of live connections and finding which server currently holds the person you’re messaging.

So the plot centers on a registry that maps each user to their connection server, a queue that routes messages with backpressure, and per-conversation sequence numbers that let any client reconnect and resync without losing or reordering a thing. Persist before you ack, dedupe on a client message id, and delivery stays correct through every crash.

How to think about it — step by step

Identify the true challenge

It isn't message volume — it's holding hundreds of millions of live connections. Why: that reframes the whole design around connection state.

Keep connections open

Clients hold a WebSocket to a connection server so the server can push. Why: chat needs the server to reach the user instantly.

Solve routing

Keep a map of user → which server holds their socket; route messages through it. Why: you must find the recipient's current server.

Guarantee order and delivery

Per-conversation sequence numbers; persist before you ack; offline queue for the disconnected. Why: messages must never drop or reorder.

Make reconnect painless

Client comes back, re-registers, and pulls everything since its last sequence. Why: phones lose signal constantly.

1 · Requirements

Functional: 1:1 and group messaging; delivered/read receipts; online presence; offline delivery (messages wait for a disconnected user); strict ordering within a conversation; optional end-to-end encryption. Non-functional: very low delivery latency, hundreds of millions of concurrent connections, high availability, durability (never lose a delivered message).

3 · Connection & protocol

HTTP request/response doesn’t fit — the server must push. Clients hold a persistent WebSocket (or MQTT) to a connection server. Every message is acknowledged (client→server→recipient→receipts) and carries a client-generated message ID for idempotency.

  • SEND {convId, clientMsgId, body} → server persists, returns {msgId, seq}.
  • Server pushes DELIVER {convId, msgId, seq, body} to recipients; clients reply ACK / READ.

4 · Data model

  • Messages (Cassandra): partition by convId, clustering key seq (monotonic per conversation) → ordered reads and easy “since seq N” sync.
  • Conversation membership: members of each conversation/group.
  • Session registry (Redis): userId => connectionServerId — where each online user is connected right now.
  • Per-user inbox / offline queue: undelivered messages for users who are offline (store-and-forward).

6 · The hard parts

  • Routing to the right server: connections are stateful, so we keep a userId→connServer map in Redis updated on connect/disconnect; the router consults it (or publishes to a per-user channel via pub/sub) to reach the recipient’s server.
  • Ordering: a monotonic per-conversation seq gives total order within a chat; clients render by seq and request “everything since my last seq” to resync.
  • Offline + delivery guarantees: store-and-forward queue per user; at-least-once delivery deduped by clientMsgId; receipts are just more messages.
  • Groups: small groups fan out to each member’s server; very large groups borrow the feed’s hybrid idea.
End-to-end encryption

With E2E (e.g. the Signal protocol) the server routes ciphertext it can’t read. That maximizes privacy but means server-side search, spam scanning, and rich previews must move to the client — a real product trade-off to name.

📍 Real example — WebSocket vs long-polling

How does a new message reach the receiver instantly? Two approaches — the old workaround and the modern answer:

// LONG-POLLING (HTTP hack): client keeps asking, server holds the request open
client → GET /messages?since=1042      (server holds it... up to 30s)
server → 200 [ ]  (nothing yet, timeout)   client immediately asks again...
server → 200 [ {msg 1043} ]  (something arrived — reply at once)
   cost: a new request/response per message, lots of overhead, near-real-time only.

// WEBSOCKET: one handshake upgrades HTTP to a persistent two-way pipe
client → GET /chat   Upgrade: websocket            (handshake, once)
server → 101 Switching Protocols                    (now a duplex channel)
   ... server PUSHES {msg 1043} the instant it arrives, no new request
   ... client sends typing/read receipts down the SAME socket

What this teaches: long-polling fakes real-time over plain HTTP — simple and firewall-friendly, but every message costs a full request and latency is bounded by the poll. A WebSocket does one Upgrade handshake (HTTP 101) and then keeps a single connection open both ways, so the server can push instantly and the client streams typing indicators back down the same pipe. The trade: you now hold millions of long-lived connections, so you need a connection/presence layer and a way to route a message to whichever server holds the recipient’s socket.

7 · Scaling

  • Connection servers scale horizontally; each holds tens-to-hundreds of thousands of sockets; a load balancer with sticky sessions assigns clients; on server loss, clients reconnect and re-register.
  • Shard the session registry and message store by user/conversation.
  • The router rides a message queue so spikes are absorbed and delivery is decoupled from ingestion.

8 · Failure & reliability

  • Reconnect & resync: client reconnects to any server, re-registers, and pulls messages since its last seq — no message lost.
  • Durability: persist before ack; replicate the store across AZs.
  • Idempotency: dedupe on clientMsgId so retries never duplicate a message.

9 · Trade-offs

  • At-least-once + dedup (simple, safe) over exactly-once (complex).
  • Stateful connections give push/low latency but make routing and failover harder — the session registry is the price.
  • E2E encryption trades server features for privacy.
Say it like a director

“The throughput is manageable; the real problem is stateful fan-in/fan-out across hundreds of millions of live sockets. I’d keep a userId-to-server registry in Redis, route through a queue for backpressure, persist-before-ack for durability, and use per-conversation sequence numbers so any client can reconnect and resync without losing or reordering a message.”

Key sub-systems

  • Connection servers — hold WebSockets.
  • Session registry — userId → server (Redis).
  • Router + queue — delivers/buffers messages.
  • Message store — per-conversation, ordered.
  • Offline queue — store-and-forward.

Bottlenecks & what to optimize next

  • Holding millions of sockets is the constraint — scale connection servers horizontally.
  • Group fan-out for large groups — borrow the feed's hybrid approach.

Common follow-up questions

How do you route a message to the right server?

A userId→connection-server map in Redis (updated on connect/disconnect); the router looks it up or publishes to a per-user channel.

How do messages stay ordered?

A monotonic per-conversation sequence number; clients render by sequence and resync 'everything since N' on reconnect.

How does E2E encryption change the design?

The server routes ciphertext it can't read, so search, previews, and spam scanning must move to the client.
