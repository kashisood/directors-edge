Security

You don't need to be a security engineer, but a director who treats security as an afterthought is a liability. The signal interviewers look for: you make the secure path the default and you can reason about risk in plain language.

On this page The security mindset Authentication vs authorization Sessions, tokens & OAuth Secrets & API keys Everyday practices Practice questions
📖 The story

Security goes wrong when it’s a gate someone runs into at the end. The story of a strong security culture is the opposite: make the secure path the easy path, so doing the right thing is the path of least resistance.

It rests on three ideas you can say in your sleep — least privilege (give the minimum), defense in depth (never trust one control), and secure-by-default (catch it early, in CI and design).

Then the specifics fall into place: identity first (authenticate) then permissions (authorize); secrets never in code; and when something breaks, contain calmly and fix the system that allowed it.

The security mindset

Three principles do most of the heavy lifting. Reach for them by name and you sound like a leader who's lived through an incident or two.

Least privilege

Every user, service, and key gets the minimum access needed — nothing more. Limits the blast radius when (not if) something is compromised.

Defense in depth

No single control is trusted alone. Layer them — network, app, data, identity — so one failure doesn't mean total compromise.

Secure by default / shift left

Make the safe path the easy path, and catch issues early in the lifecycle (in CI, in design) rather than after release. Security is everyone's job, enabled by the platform.

Zero Trust in one line

"Never trust, always verify." Don't assume anything inside the network is safe — authenticate and authorize every request regardless of where it came from. The modern replacement for the old "hard shell, soft center" perimeter model.

Authentication vs. authorization

The single most common security question, and an easy one to fumble under pressure. Keep them crisp.

AuthenticationAuthorization
QuestionWho are you?What can you do?
HappensFirstAfter authentication
ExamplesPassword, MFA, biometrics, SSORBAC roles, ABAC policies, OAuth scopes
Visible to user?Yes (you log in)Usually invisible (access just works or is denied)

RBAC — Role-Based Access Control

Permissions attach to roles (admin, editor, viewer); users get roles. Simple, auditable, the common default. Can sprawl into "role explosion" at scale.

ABAC — Attribute-Based Access Control

Access decided by attributes (department, region, time, resource owner). More flexible and fine-grained than RBAC, but more complex to reason about and audit.

Say it like a director

"Authentication is verifying who you are; authorization is deciding what you're allowed to do — identity first, permissions second. I'd standardize authentication on SSO with MFA so we're not managing passwords, and start authorization with RBAC for clarity and auditability, moving to attribute-based rules only where the business genuinely needs that granularity. The principle underneath both is least privilege."

📍 Real example — a JWT, decoded

A JWT (JSON Web Token) is just three Base64URL chunks joined by dots: header.payload.signature. Here is a real one:

eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiJ1c3JfNDgyMSIsInJvbGUiOiJhZG1pbiIsImlhdCI6MTczODMxNjgwMCwiZXhwIjoxNzM4MzIwNDAwfQ.3pErEaB7Qe1n_kZ2yq8VtJ0r9sXc1d2e3f4g5h6i7k

Split on the dots and Base64-decode the first two parts — they are plain JSON, not encrypted:

// header
{ "alg": "HS256", "typ": "JWT" }

// payload (the “claims”)
{ "sub": "usr_4821", "role": "admin", "iat": 1738316800, "exp": 1738320400 }

// signature = HMAC-SHA256( base64url(header) + "." + base64url(payload), SECRET )

What this teaches: anyone can read the header and payload (paste them into jwt.io), so never put secrets in a JWT. The signature is what matters — the server recomputes it with its secret to confirm nobody tampered with the claims. Always check exp (expiry), keep lifetimes short, and remember you can’t easily revoke a JWT before it expires (hence refresh tokens).

Sessions, tokens & OAuth

Know the vocabulary well enough to govern decisions, even if the team implements the details.

  • Session-based auth: server stores a session, client holds a session ID cookie. Simple, easy to revoke, but stateful — harder to scale horizontally.
  • Token-based (JWT): a signed token carries identity/claims; the server can verify it without storing state. Scales well and is stateless, but revoking a token before it expires is harder — keep lifetimes short and pair with refresh tokens.
  • OAuth 2.0: a delegation framework — "let app A act on my behalf at service B" without sharing your password. The basis of "Sign in with Google" and most third-party API access.
  • OpenID Connect (OIDC): an identity layer on top of OAuth 2.0 — OAuth handles authorization/delegation, OIDC adds standardized authentication (who the user is).
  • MFA: multiple factors — something you know, have, or are. The single highest-leverage control against account takeover; mandate it for anything sensitive.
Common confusion to get right

OAuth is fundamentally about authorization/delegation, not authentication — that's what OIDC adds on top. Mixing these up is a classic tell that someone has used the libraries but not understood them. As a director, you don't implement it, but you should catch the team conflating the two.

Secrets & API key management

Secrets are credentials your systems use: API keys, database passwords, private keys, tokens. Mishandled secrets are one of the most common and most damaging breaches — and one a director is expected to have firm opinions about.

The cardinal sin

Never hard-code secrets in source code or commit them to Git. Once a secret hits the repo history it's effectively public forever, even if you delete it later — it must be rotated. This is the number-one thing to have a zero-tolerance policy on.

Where secrets should live

In a dedicated secrets manager (HashiCorp Vault, AWS Secrets Manager, Azure Key Vault, etc.) or injected as environment variables at runtime — never in code, config files in the repo, or chat messages.

Rotation

Secrets should expire and rotate automatically on a schedule, and immediately if exposure is suspected. Short-lived, frequently rotated credentials limit the value of a leaked one.

Least-privilege keys

Scope each API key to only what it needs — read-only if it only reads, one service per key — so a single leaked key can't do much. Avoid one all-powerful "god key."

Detection & encryption

Run secret-scanning in CI to catch accidental commits, encrypt secrets at rest and in transit (TLS everywhere), and audit who accessed what. Assume leaks will happen and limit their impact.

Say it like a director

"My baseline policy is: no secret ever lives in source control. We use a managed secrets store, inject credentials at runtime, scope every key to least privilege, and rotate automatically — with immediate rotation if anything's suspected leaked. I also wire secret-scanning into CI so a mistake is caught before merge, not after a breach. The goal is that the secure way is also the most convenient way, so engineers don't route around it."

📍 Real example — secrets & logs

✗ Secret hard-coded

DB_PASSWORD = "S3cr3t!" // in code — now in git forever

✓ Injected at runtime

DB_PASSWORD = vault.read("db/prod").password

Same for logs: never log tokens, passwords, API keys, PII or card data. Log user_id=4821 action=login ok, not the password or the full request body. Enforce with a redaction filter + secret-scanning in CI — the most common leak is a developer logging the whole payload.

📍 Real example — OAuth2 Authorization Code + PKCE

“Sign in with Google” uses the OAuth2 authorization-code flow. PKCE (“pixie”) hardens it for mobile/SPA apps that can’t keep a secret. The app first invents a random code_verifier and sends only its hash:

1. app makes  code_verifier = random  ;  code_challenge = base64url(SHA256(verifier))

2. browser → accounts.google.com/o/oauth2/v2/auth
      ?client_id=...&redirect_uri=...&scope=email
      &code_challenge=XYZ&code_challenge_method=S256

3. user logs in + consents  →  Google redirects back:
      https://app/callback?code=AUTH_CODE_ONE_TIME

4. app → POST /token   { code=AUTH_CODE, code_verifier=ORIGINAL_RANDOM }
      Google checks  SHA256(verifier) == code_challenge   →  issues tokens:
      { access_token, refresh_token, id_token (a JWT), expires_in: 3600 }

What this teaches: the password is only ever entered at Google — your app never sees it, it just receives a short-lived, scoped access_token. The one-time code travels through the browser redirect, but it’s useless to an attacker without the matching code_verifier that never left the app — that’s what PKCE protects against (an intercepted code). Always request the narrowest scope, keep access tokens short-lived, and store the refresh_token securely to get new ones without re-prompting.

Everyday practices a director champions

  • Encrypt in transit and at rest — TLS everywhere, encrypted storage by default.
  • Validate and sanitize all input — to prevent injection (SQL injection, XSS); use parameterized queries.
  • Patch and manage dependencies — most breaches exploit known, unpatched vulnerabilities; automate dependency scanning.
  • Audit logging — record who did what, so you can investigate and prove compliance.
  • Threat modeling — for significant features, ask "how would an attacker abuse this?" during design.
  • Incident response plan — know in advance who decides, who communicates, and how you contain — rehearsed, not improvised.
  • OWASP Top 10 — the canonical list of the most common web vulnerabilities; a useful shared reference for the team.

Practice questions

What's the difference between authentication and authorization?Core

Authentication answers "who are you?" — proving identity through something like a password, MFA, or SSO. Authorization answers "what are you allowed to do?" — checking permissions once identity is established. Authentication always comes first. A simple analogy: authentication is showing your ID to get into the building; authorization is whether your badge opens a particular door. Underpinning both is least privilege — grant the minimum access required.

How does your team manage secrets and API keys?Core

The non-negotiable is that secrets never live in source control — once a key is in Git history it's compromised and must be rotated. We keep secrets in a managed store like Vault or a cloud secrets manager, inject them at runtime rather than baking them into images, scope every key to least privilege, and rotate on a schedule plus immediately on any suspected exposure. I also put secret-scanning in CI so an accidental commit is blocked before merge. The principle is to make the secure path the path of least resistance so people don't work around it.

A developer accidentally committed an API key to a public repo. What now?Trap

Treat the key as compromised the moment it's pushed — deleting the commit doesn't help, because it's in history and likely already scraped. So: rotate/revoke the key immediately, check logs for any unauthorized use, and assess blast radius based on what that key could access. Then the systemic fix: add secret-scanning to CI and pre-commit hooks so it can't happen again, and run a blameless review — the goal is fixing the system that allowed it, not punishing the person, or people will hide the next mistake. Calm, fast containment plus a process fix is the director's signature here.

Is OAuth authentication or authorization?Deep

OAuth 2.0 is fundamentally an authorization/delegation framework — it lets one application act on a user's behalf at another service without sharing the password, via scoped access tokens. It is not an authentication protocol by itself. The authentication layer is OpenID Connect (OIDC), which sits on top of OAuth and standardizes proving who the user is. People commonly conflate them because "Sign in with Google" uses both. As a leader I don't implement the flows, but I do expect the team to use OIDC for identity and OAuth for delegated access, and to know the difference.

How do you build security into the development process without slowing teams down?Deep

By making security a paved road rather than a gate. I shift left — automated checks in CI for dependencies, secrets, and common vulnerabilities — so issues surface in minutes, not in a review weeks later. I give teams secure defaults and reusable components so the easy path is the safe one, threat-model the high-risk features at design time, and reserve human security review for genuinely high-stakes changes. The mindset is guardrails, not gates: security enables the team to move fast safely instead of standing between them and production.

What’s the difference between encryption in transit and at rest?Core

Encryption in transit protects data while it moves across a network — that’s TLS/HTTPS on every connection, so an attacker who intercepts traffic sees gibberish. Encryption at rest protects data while it’s stored — on disks, in databases and backups — so a stolen drive or backup is useless without the keys. You want both, by default, plus sound key management. As a leader my baseline expectation is TLS everywhere and encrypted storage as a default setting, not a special request — and that we manage and rotate the keys properly, since encryption is only as strong as key handling.

What does least privilege look like in practice?Core

It means every user, service, and credential gets exactly the access it needs and nothing more. In practice: scoped API keys (read-only if it only reads), narrow IAM roles instead of broad admin, time-bound and just-in-time access for sensitive operations, and regular access reviews to remove permissions people no longer need. It limits the blast radius — when something is compromised, and eventually something will be, the damage is contained. I treat ‘does this need that much access?’ as a routine review question, and I’m especially wary of long-lived, all-powerful credentials.

How do you lead a security incident as the manager?Deep

My job in an incident is to bring calm, structure, and speed, not to debug personally. I make sure there’s a clear incident commander, contain first (revoke credentials, isolate affected systems) before chasing root cause, and assess the blast radius from logs. I handle communication — keeping stakeholders and, where appropriate, customers honestly informed — so engineers can focus. Afterward I drive a blameless post-mortem that fixes the system that allowed the incident, not the individual, because blame just makes the next problem get hidden. The reflexes are: contain fast, communicate honestly, learn systemically.

How do you handle compliance like SOC 2 or GDPR without killing velocity?Deep

I treat compliance as something we build into the platform rather than bolt on at audit time. That means automated evidence and controls (access reviews, change management, encryption, logging) baked into our normal workflows, and secure-by-default templates so teams meet requirements just by using the paved road. I bring security and legal in early on significant features so we design for privacy and data handling up front instead of reworking later. Framed well, compliance is largely the same hygiene that good engineering already wants — the trick is automating the evidence so it doesn’t become a manual tax on every team.

Security wants to block a launch over a medium-risk finding. The business wants to ship. What do you do?Trap

I’d make it a clear-eyed risk decision rather than a turf fight. I’d get the actual risk articulated — exploitability, blast radius, what data is exposed — and the cost of delay, then put both in front of the people accountable for the trade-off. Often there’s a middle path: ship to a limited cohort, add a compensating control or monitoring, and commit to a dated fix. If it’s genuinely high impact I’ll back security and hold the launch; if it’s truly low and reversible, we may accept the risk explicitly with an owner and a deadline. What I won’t do is let it be decided by whoever pushes hardest — it’s a documented, owned decision.
