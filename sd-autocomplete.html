📖 The story

Autocomplete has to feel like magic — suggestions appear as you type, in tens of milliseconds. You can’t compute that per keystroke from scratch; you precompute it.

So an offline job mines query logs into a ranked prefix structure (a trie), which is served from memory behind an edge cache. Reads are instant; freshness comes from periodic rebuilds.

How to think about it — step by step

Set the brutal latency bar

Suggestions must appear as you type — tens of milliseconds. Why: that rules out computing anything from scratch per keystroke.

Precompute the answers

Offline, mine query logs into a ranked list of top completions for each prefix. Why: at request time you just look up, you don't search.

Serve from memory + edge

Hold the structure (a trie) in memory behind an edge cache. Why: memory + nearby cache is how you hit the latency budget.

Refresh on a schedule

Rebuild periodically so trending terms appear. Why: full real-time freshness isn't worth the cost.

Name the trade-off

Personalized suggestions are more relevant but can't be shared/cached. Why: freshness and personalization both fight latency.

1 · Requirements

Functional: return the top few completions for a prefix, ranked by popularity, as the user types. Non-functional: p99 latency in the tens of ms; huge query volume; reasonable freshness.

2 · Capacity

Read-dominated; the data (popular prefixes) is small enough to hold in memory and cache at the edge.

4 · The hard parts

  • Latency budget: precompute top-k per prefix so a query is a memory lookup, not a search.
  • Ranking: weight by frequency/recency; personalize cautiously (kills cacheability).
  • Freshness: trending terms appear via periodic rebuilds; full real-time is expensive.

5 · Trade-offs

  • Freshness vs latency (precomputed cached vs real-time updates).
  • Personalization (relevant) vs cacheability (fast, shared).

Key sub-systems

  • Log aggregator — mines query frequencies offline.
  • Trie builder — precomputes top-k per prefix.
  • Suggest service — in-memory lookups.
  • Edge cache — hot prefixes.
  • Ranking — frequency + recency signals.

Bottlenecks & what to optimize next

  • The latency budget is the constraint — precompute so a query is a memory lookup, not a search.
  • Freshness costs — rebuild the trie periodically rather than per query.

Common follow-up questions

How is it so fast?

Top completions per prefix are precomputed offline and held in memory behind an edge cache, so a request is a lookup, not a computation.

How do trending terms appear?

Rebuild/merge the trie periodically from recent query logs; full real-time updating is rarely worth the cost.

How do you stop offensive/poisoned suggestions?

Blocklists, frequency thresholds, and review of newly trending terms before surfacing them.
