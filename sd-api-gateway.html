📖 The story

As soon as you have many services, you don’t want each one re-implementing auth, rate limiting, and TLS. The gateway is the single front door that handles those cross-cutting concerns once.

The catch: being on every request, it must be blazing fast and never the thing that takes the whole API down — so it’s stateless, horizontally scaled, and caches what it can.

How to think about it — step by step

Spot the repeated work

Every service would otherwise re-implement auth, rate limiting, TLS. Why: that's duplication and inconsistency — a job for one shared layer.

Make it the single front door

Route, authenticate, rate-limit, and observe in one place, then forward to the right service. Why: services stay clean and focused on business logic.

Keep it stateless

No per-user state on the gateway itself. Why: so you can run many copies behind a load balancer.

Protect against the obvious risk

It's on every request, so it must be fast and never a single point of failure. Why: if the gateway is slow or down, everything is.

Name the trade-off

One consistent policy layer vs a shared dependency everything relies on. Why: centralization helps and creates risk — say both.

1 · Requirements

Functional: one entry point that authenticates, authorizes, rate-limits, routes, and optionally aggregates calls to backend services. Non-functional: very low added latency, high availability — it’s on every request.

3 · The hard parts

  • Central choke point: it must add minimal latency and never be a single point of failure — run many stateless instances, cache auth decisions.
  • Cross-cutting concerns: auth, rate limiting, retries, circuit-breaking, request/response transformation live here so services stay clean.
  • Routing: a config-driven route table (path/host → service), versioned.

4 · Trade-offs

  • Centralizing concerns (clean services, consistent policy) vs a shared dependency everything relies on.
  • Aggregation at the gateway (fewer client round-trips) vs gateway complexity (BFF pattern can help).

Key sub-systems

  • TLS termination — ends HTTPS at the edge.
  • AuthN/AuthZ — validate JWT/keys with the IdP.
  • Rate limiting — protect backends.
  • Router — path/host → service; optional aggregation.
  • Observability — metrics, traces, logs.

Bottlenecks & what to optimize next

  • The gateway is on every request — keep it stateless and horizontally scaled so it's never a SPOF.
  • Auth checks add latency — cache validated tokens/decisions briefly.

Common follow-up questions

Doesn't a single gateway become a bottleneck?

It's stateless, so you run many instances behind a load balancer; cache auth decisions and keep logic thin to minimize added latency.

Where do retries and circuit-breaking live?

At the gateway (or a service mesh), so cross-cutting resilience is consistent and services stay simple.

Gateway vs service mesh?

Gateway handles north-south (client→system) traffic; a mesh handles east-west (service→service). Large systems use both.

🧠 Remember it

API gateway = one stateless front door for cross-cutting concerns.

  • AuthN, rate limiting, routing, aggregation, observability in one place.
  • Stateless + horizontally scaled so it’s never a SPOF.
  • Keeps backend services clean; mind the shared-dependency risk.
