📖 The story

A matching engine is the rare system where microseconds and absolute fairness both matter, and a wrong fill is catastrophic. Determinism beats raw parallelism here.

So the order book lives in memory and is matched single-threaded per instrument (so the order of events is exact and reproducible), every order is appended to a sequenced log for durability and replay, and market data fans out separately.

How to think about it — step by step

Prize determinism over parallelism

Fairness and reproducible order matter more than raw speed-via-threads. Why: it leads you to single-threaded matching.

Match in memory, single-threaded

One thread per instrument processes the order book. Why: a strict, reproducible sequence is essential for fairness and audit.

Make it durable anyway

Append every sequenced order to a fast log. Why: you can rebuild the book after a crash without losing trades.

Chase microseconds

In-memory structures, kernel-bypass networking, co-location, no GC pauses. Why: latency is the product.

Parallelize the right way

Across instruments, not within one. Why: keeps each book deterministic while still scaling.

1 · Requirements

3 · The hard parts

  • Determinism: single-threaded matching per instrument gives a strict, reproducible order — essential for fairness and audit.
  • Latency: in-memory data structures, kernel-bypass networking, co-location; avoid GC pauses.
  • Durability: append every sequenced event to a fast log so the book can be rebuilt after a crash.

4 · Trade-offs

  • In-memory speed vs durability — reconciled by a fast append-only log.
  • Single-threaded matching (deterministic, fair) vs parallelism (you parallelize across instruments, not within one).

Key sub-systems

  • Order gateway — sequences incoming orders.
  • Matching engine — single-threaded in-memory order book.
  • Event log — sequenced, durable.
  • Market data — fan-out to subscribers.

Bottlenecks & what to optimize next

  • Latency is the product — in-memory, kernel-bypass networking, no GC pauses, co-location.

Common follow-up questions

Why single-threaded matching?

Determinism and fairness — a strict, reproducible order of events. You parallelize across instruments, not within one book.

How is it durable if it's in memory?

Every sequenced order/trade is appended to a fast log, so the book can be rebuilt after a crash with no lost trades.

How do you achieve microsecond latency?

In-memory structures, kernel-bypass networking, avoiding GC pauses, and co-locating with the exchange.

🧠 Remember it

Matching engine = single-threaded in-memory order book + sequenced log.

  • Single-threaded per instrument = deterministic, fair ordering.
  • Append-only sequenced log for durability/replay.
  • Parallelize across instruments, not within; co-locate for latency.
