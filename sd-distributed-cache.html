Design a Distributed Cache (Redis-like)

A deep-dive into the in-memory cache tier: consistent hashing, replication, eviction, and the classic hot-key and stampede problems.

📖 The story

A cache exists to not redo expensive work — but at scale one machine isn’t enough, so you spread keys across many. The art is spreading them so that adding or losing a node barely disturbs the rest.

That’s consistent hashing. Layer on replication for availability, eviction to bound memory, and answers for the two classic villains — hot keys and cache stampedes — and you have a production cache.

How to think about it — step by step

Know why the cache exists

To avoid redoing expensive work (DB queries). Why: keeps the goal — speed — front of mind.

Spread keys across machines

Use consistent hashing so each key maps to a node, and adding/removing a node only moves a few keys. Why: naive hashing would reshuffle everything when the cluster changes.

Bound the memory

Evict with LRU and set TTLs. Why: RAM is finite; old, cold data should make room.

Plan for the nasty cases

Hot keys overload one node (replicate them); a stampede happens when a popular key expires and everyone hits the DB at once (stagger TTLs, let one request rebuild). Why: these are the real production failures.

Decide invalidation

TTL, write-through, or delete-on-update. Why: stale cache = wrong answers, so you must choose deliberately.

1 · Requirements

Functional: sub-millisecond get/set fronting a slower datastore. Non-functional: high throughput, horizontal scale, high availability.

2 · Architecture

📍 Real example — the consistent-hashing ring

Say you cache by hash(key) % N across N=4 servers. Add one server and almost every key now maps somewhere new — the whole cache misses at once (a “cache stampede”). Consistent hashing fixes this by placing servers and keys on the same 0–232 ring; a key belongs to the next server clockwise:

ring position (hash, mod 360 for readability)

   node-A @ 25 °        key "cart:42"  @ 200°  → next clockwise = node-C
   node-B @ 120°        key "user:7"   @ 30 °  → next clockwise = node-B
   node-C @ 210°        key "sess:9"   @ 350°  → wraps to     = node-A
   node-D @ 300°

add node-E @ 130°  →  ONLY keys between 120° and 130° move (B→E). Everything else stays put.

Picture that diagram as a clock face. Servers sit around the ring — node A near 25 degrees, B at 120, C at 210, D at 300 — and every key is placed on the same ring and handed to the next server clockwise. A key at 200 degrees goes to node C; a key near the top wraps around to node A. When you add a new node at 130 degrees, only the keys between 120 and 130 move to it — every other key stays exactly where it was.

What this teaches: adding or removing a node only re-maps the keys in one arc of the ring (~1/N of them), not all of them — so scaling doesn’t nuke the cache. In practice each physical node is placed at many points on the ring (“virtual nodes”) so load spreads evenly and one node leaving doesn’t dump its whole load on a single neighbour. This is exactly how Redis Cluster, DynamoDB, and Cassandra decide where data lives.

3 · The hard parts

  • Consistent hashing + virtual nodes spread keys evenly and minimize reshuffle on membership change.
  • Eviction: LRU/LFU with TTLs to bound memory.
  • Hot keys: replicate or add a small local cache; stampede: stagger TTLs + single-flight recomputation.
  • Invalidation: TTL, write-through, or explicit delete on update.

4 · Trade-offs

  • Replication = availability + read scale vs memory cost + consistency lag.
  • Cache-aside (simple, possible staleness) vs write-through (fresher, slower writes).

Key sub-systems

  • Client library — cache-aside logic.
  • Hash ring — consistent hashing maps keys to nodes.
  • Cache nodes — in-memory store + replicas.
  • Eviction — LRU/LFU + TTLs.
  • Membership — gossip/coordination for node changes.

Bottlenecks & what to optimize next

  • Hot keys overload one node — replicate hot keys or add a tiny client-side cache.
  • Mass expiry causes a stampede — jitter TTLs and let one request rebuild (single-flight).

Common follow-up questions

Why consistent hashing instead of modulo?

Modulo reshuffles almost every key when the node count changes; consistent hashing moves only a small fraction, so the cache survives scaling.

How do you prevent a cache stampede?

Stagger TTLs with jitter and use single-flight so only one request recomputes a hot key while others wait.

How do you keep cache and DB consistent?

Pick a policy: TTL (simple, brief staleness), write-through (fresher), or explicit invalidation on write.

🧠 Remember it

Distributed cache = consistent hashing + replication + eviction; beware hot keys & stampedes.

  • Consistent hashing (virtual nodes) minimizes reshuffle.
  • Cache-aside; TTL/write-through for invalidation.
  • Hot keys → replicate; stampede → jitter TTL + single-flight.
