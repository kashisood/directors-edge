📖 The story

You can’t run what you can’t see, so every service emits a firehose of metrics, logs, and traces. The trouble is cardinality — every unique combination of labels is its own series, and they explode.

So a metrics system ingests time-series at scale, stores recent data at high resolution and downsamples older data, and layers a query engine and alerting on top — spending storage where it earns its keep.

How to think about it — step by step

Start from the purpose

You can't operate what you can't see. Why: the system must answer new questions, not just known ones.

Name the cost driver: cardinality

Every unique combination of labels is its own series, and they explode. Why: this dominates storage and query cost.

Tier the data

Keep recent data high-resolution; downsample old data. Why: you rarely need per-second detail from last month.

Add query + alerting

A query layer powers dashboards; an engine evaluates alert rules continuously. Why: seeing and being warned are different jobs.

Name the trade-off

Resolution/retention vs cost; cardinality limits vs granularity. Why: observability budgets are real.

3 · The hard parts

  • Cardinality: each unique label-set is a series; high cardinality explodes storage and query cost — limit labels, aggregate.
  • Retention tiers: keep recent data high-resolution, downsample (roll up) older data to control cost.
  • Alerting: evaluate rules continuously with minimal lag; dedupe and route alerts.

4 · Trade-offs

  • Resolution/retention vs storage cost (downsample old data).
  • Cardinality limits vs query granularity.

Key sub-systems

  • Agents — push metrics/logs/traces.
  • Ingestion tier — scalable intake.
  • TSDB — time-series + rollups.
  • Query layer — dashboards.
  • Alerting engine — rules → notify.

Bottlenecks & what to optimize next

  • Cardinality explodes storage/query cost — limit labels and aggregate.
  • Old high-res data is expensive — downsample with retention tiers.

Common follow-up questions

Why is cardinality the cost driver?

Each unique combination of metric labels is its own time series; high cardinality multiplies storage and slows queries.

How do you control storage cost?

Keep recent data high-resolution and downsample older data into rollups via retention tiers.

Metrics vs logs vs traces?

Metrics = cheap aggregates ('is it down?'); logs = detailed events; traces = a request's path across services ('why is this slow?').

🧠 Remember it

Observability = time-series ingest + downsampling/retention + alerting.

  • High cardinality is the cost driver — limit labels.
  • Recent = high-res; old = downsampled rollups.
  • Query layer + alerting on top of the TSDB.
