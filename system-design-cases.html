System Design II — Case Studies

Thirty real system-design questions, the kind you get at Google and other big-tech loops. Each is walked through at a leader’s altitude: clarify the requirements and scale, sketch the core design, name the hardest part, and state the key trade-off. Don’t memorize them — absorb the pattern so you can reason through any prompt.

📖 The story

After enough designs you notice they’re the same few stories retold. Read-heavy? Precompute and cache. Write-heavy and moving? Partition and index. Money or seats? Be strongly consistent right there and nowhere else.

So don’t memorize thirty answers — collect a handful of patterns and learn which prompt triggers which. The six flagship deep-dives show the patterns in full; the rest are reps to recognize them fast.

When a new prompt lands, you won’t reach for a memorized solution — you’ll recognize the shape and reason from the pattern. That’s what separates a senior designer from a crammer.

The method — same four moves every time

Whatever the prompt, walk these out loud. The interviewer is scoring your structured thinking far more than any single component choice.

  • Clarify & scale: who uses it, read vs write ratio, how many users/QPS, latency and consistency needs. Never design blind.
  • Core design: the main components and data flow — API, storage, the few services that matter.
  • Hardest part: name the real bottleneck or risk and how you’d tackle it. This is where seniority shows.
  • Key trade-off: what you’re optimizing for and what you’re giving up (consistency vs availability, cost vs latency, simplicity vs scale).
Say it like a director

"Before I draw anything: what scale are we solving for, and are we optimizing for latency, cost, or consistency? The right design for a startup is the wrong one for a billion users, so let me anchor on that first."

1 · Design a URL shortener (TinyURL)
  • 🔍 Full deep-dive (with diagrams) →
  • Scale: Read-heavy, ~100:1 reads:writes; e.g. 100M new links/day (~1.2K writes/s) vs tens of thousands of redirects/s.
  • Core design: Write API returns a short code; a KV store (DynamoDB / Redis) maps code → longUrl; redirects served from cache + CDN; stateless app tier behind an LB; HTTP 301/302.
  • Data model: code (PK, ~7 chars base62), longUrl, ownerId, createdAt, expiry.
  • Hardest part: Generating short unique codes with no coordination — base62 of a Snowflake/counter range, or hash(longUrl) with collision retry.
  • Key trade-off: Counter (sequential, guessable) vs hash (no coordination, collisions); 301 (cacheable, no analytics) vs 302 (counts every hit).
2 · Design Pastebin
  • 🔍 Full deep-dive (with diagrams) →
  • Scale: Read-heavy; large text blobs; optional TTL expiry.
  • Core design: Blob in object storage (S3); metadata + short key in a DB/KV; CDN for popular reads; background expiry sweeper.
  • Data model: pasteId (PK), blobUrl, language, createdAt, expireAt, visibility.
  • Hardest part: Cheaply storing many large blobs while keeping lookups fast and supporting expiry without scanning everything.
  • Key trade-off: Blob store (cheap, scalable) vs DB (simple, pricey at size) — DB holds only metadata; TTL precision vs sweep cost.
3 · Design a distributed unique ID generator
  • 🔍 Full deep-dive (with diagrams) →
  • Scale: Millions of IDs/s across many hosts; ideally time-sortable; 64-bit.
  • Core design: Snowflake: [timestamp | machineId | sequence] generated locally, zero coordination; machineId from a registry.
  • Data model: 64-bit ID = 41 bits ms-timestamp + 10 bits machine + 12 bits per-ms sequence.
  • Hardest part: Avoiding collisions and a central bottleneck while staying roughly sortable; clock skew / NTP regression.
  • Key trade-off: UUID (no coordination, unsortable, 128-bit) vs DB auto-increment (sortable, single bottleneck) vs Snowflake (balanced, clock-dependent).
4 · Design a feature-flag / config service
  • 🔍 Full deep-dive (with diagrams) →
  • Scale: Read-heavy from every service instance; rare writes; changes must propagate in seconds.
  • Core design: Small flag store + API; SDK in each service caches flags with short TTL or a streaming/push channel (SSE / pub-sub) for instant invalidation.
  • Data model: flagKey, value/variant, targeting rules (by user %, region), version, updatedAt.
  • Hardest part: Propagating a change to thousands of instances fast without hammering the store on every request.
  • Key trade-off: Polling (simple, laggy, load) vs push/streaming (instant, more moving parts); cache aggressively and tolerate brief staleness.
5 · Design a rate limiter
  • 🔍 Full deep-dive (with diagrams) →
  • Scale: ~1M checks/s across the gateway fleet; must add <1ms; global (not per-node) limit.
  • Core design: Token bucket per key in shared Redis; check-and-decrement as one atomic Lua script; enforced at the API gateway; reject with 429 + Retry-After.
  • Data model: bucket:{key} => {tokens, lastRefillTs} in Redis; rules per route.
  • Hardest part: Atomic distributed counting under concurrency without a per-request network hop becoming the bottleneck.
  • Key trade-off: Centralized Redis (accurate, +hop) vs local bucket + periodic sync (fast, approximate); fail-open vs fail-closed.
6 · Design a distributed cache (Redis-like)
  • 🔍 Full deep-dive (with diagrams) →
  • Scale: Sub-ms reads fronting a slow store; 100s of GB–TBs in RAM; high throughput.
  • Core design: Shard keys via consistent hashing; replicate each shard (primary + replicas); LRU eviction; cache-aside in the app.
  • Data model: Opaque key → serialized value + TTL; slots mapped to nodes by hash ring.
  • Hardest part: Hot keys overloading one shard; invalidation; thundering-herd on expiry.
  • Key trade-off: More replicas = availability + read scale but memory cost + replication lag; consistent hashing minimizes reshuffle on membership change.
7 · Design an API gateway
  • 🔍 Full deep-dive (with diagrams) →
  • Scale: Fronts all traffic (100Ks req/s); must add minimal latency and be HA.
  • Core design: Stateless gateway tier behind an L4/L7 LB doing TLS termination, authN, rate limiting, routing/aggregation, and emitting metrics/traces.
  • Data model: Route table (path → service), auth keys/JWT validation config, per-route limits.
  • Hardest part: It’s a central choke point — one bug or slowdown hits everything.
  • Key trade-off: Centralizing cross-cutting concerns (clean services) vs a single dependency (mitigate with horizontal scale, caching, graceful degradation).
8 · Design a social news feed (Twitter / X)
  • 🔍 Full deep-dive (with diagrams) →
  • Scale: ~300M DAU, 50:1 read:write; ~35K posts/s peak, ~1M timeline reads/s peak.
  • Core design: Fan-out on write: Post service persists to Cassandra, then async workers push tweetIds into each follower’s Redis timeline list; read = fetch one list + hydrate.
  • Data model: tweets(tweetId PK, authorId, text, mediaIds, ts); follows graph; timeline:{userId} => [tweetId…] capped ~800.
  • Hardest part: Celebrity fan-out (one post → 100M writes) — solved with a hybrid (pull-on-read for high-follower accounts).
  • Key trade-off: Extra write work for cheap reads; eventual consistency for availability; hybrid complexity to avoid write storms.
9 · Design photo sharing (Instagram)
  • 🔍 Full deep-dive (with diagrams) →
  • Scale: Read-heavy image serving + feed; upload spikes; global.
  • Core design: Images to object storage + CDN; metadata in a sharded DB; feed via the news-feed pattern; async pipeline makes thumbnails/sizes on upload.
  • Data model: media(id, ownerId, blobUrls{sizes}, caption, ts); follows; timeline cache.
  • Hardest part: Serving images globally at low latency and absorbing upload bursts.
  • Key trade-off: Pre-generate sizes (storage cost, fast serve) vs on-the-fly (compute, latency); CDN cost vs origin load.
10 · Design search autocomplete (typeahead)
  • 🔍 Full deep-dive (with diagrams) →
  • Scale: Must feel instant (p99 < ~50ms); huge query volume; ranked by popularity.
  • Core design: Precompute a ranked prefix structure (trie / FST) offline from query logs; serve from in-memory services behind heavy edge caching; update async.
  • Data model: trie nodes → top-k completions per prefix with frequency scores.
  • Hardest part: Hitting a tight latency budget while keeping suggestions fresh and well-ranked.
  • Key trade-off: Freshness vs latency — precomputed cached results are fast but stale; real-time updates add cost.
11 · Design a notification system
  • 🔍 Full deep-dive (with diagrams) →
  • Scale: Multi-channel (push/email/SMS/in-app); millions/min; seconds of delay OK.
  • Core design: Producers publish events to a queue (Kafka); stateless workers apply preferences + rate limits, render templates, hand off to provider adapters (APNs/FCM, SES, Twilio).
  • Data model: notification(id, userId, channel, template, payload, status); user prefs; dedupe keys.
  • Hardest part: Reliable delivery despite provider failures, with dedup and quiet-hours/preferences.
  • Key trade-off: At-least-once + idempotent dedup (simple) vs exactly-once (complex); per-provider retry/backoff.
12 · Design a leaderboard
  • 🔍 Full deep-dive (with diagrams) →
  • Scale: Real-time ranks for up to 10s of millions of players; constant score updates.
  • Core design: Sorted set (Redis ZSET) keyed by score for top-N and rank queries; shard by game/segment; periodic snapshots to a DB for durability.
  • Data model: zset board:{id} => member:score; archive table for history.
  • Hardest part: Exact rank for tens of millions updated constantly is expensive.
  • Key trade-off: Exact ranking (costly at scale) vs approximate (bucketing/percentiles, cheap); memory vs durability.
13 · Design a distributed job scheduler (cron at scale)
  • 🔍 Full deep-dive (with diagrams) →
  • Scale: Millions of scheduled jobs; run within seconds of due time; no duplicate runs.
  • Core design: Jobs stored with nextRunAt, partitioned; workers claim due jobs via a lease/lock (DB row lock or Redis); execute with retries + dead-letter; idempotent completion.
  • Data model: job(id, schedule, nextRunAt, status, ownerLease, attempts).
  • Hardest part: Guaranteeing each job runs despite worker crashes, without running twice.
  • Key trade-off: At-least-once + idempotency (safe, simple) vs exactly-once (hard); polling interval vs timeliness.
14 · Design an email service (Gmail-scale)
  • 🔍 Full deep-dive (with diagrams) →
  • Scale: Billions of mailboxes; massive send + receive; full-text search.
  • Core design: SMTP ingestion tier; bodies in object storage with a per-user search index (inverted index / search cluster); sending via a queue with retries, backoff, reputation; async spam filtering.
  • Data model: message(id, mailboxId, headers, blobRef, labels, ts); per-user index shards.
  • Hardest part: Storage at scale, fast search across huge mailboxes, and deliverability/spam.
  • Key trade-off: Index everything (fast search, storage/compute cost) vs lazy; send throughput vs protecting sender reputation.
15 · Design a chat system (WhatsApp / Messenger)
  • 🔍 Full deep-dive (with diagrams) →
  • Scale: ~2B users, ~100B msgs/day (~1.2M/s); 100s of millions of concurrent connections.
  • Core design: Persistent WebSocket to connection servers; a router + queue deliver via a userId→server registry (Redis); store-and-forward for offline; per-conversation seq for ordering.
  • Data model: messages(convId PK, seq clustering, senderId, body, ts); membership; session: userId→connServer; offline queue.
  • Hardest part: Holding 100s of millions of stateful sockets and routing a message to the recipient’s current server.
  • Key trade-off: At-least-once + dedup over exactly-once; stateful connections (push, low latency) vs harder routing/failover; E2E encryption vs server features.
16 · Design a web crawler
  • 🔍 Read the full deep-dive (with diagrams) →
  • Scale: Crawl billions of pages; polite per-domain; dedup at billions of URLs; stay fresh.
  • Core design: URL frontier (priority queue) → distributed fetchers (respect robots.txt, per-domain rate limit) → parser extracts links → back to frontier; seen-set via bloom filter; content to storage.
  • Data model: frontier (url, priority, host); seen bloom filter; doc store (url, contentHash, fetchedAt).
  • Hardest part: Scale + politeness + dedup + avoiding crawler traps and duplicate content.
  • Key trade-off: Crawl freshness vs cost — recrawl often (fresh, wasteful) vs prioritize by importance/change rate.
17 · Design video streaming (YouTube / Netflix)
  • 🔍 Full deep-dive (with diagrams) →
  • Scale: ~1B watch-hours/day; hundreds of upload-hours/min; petabytes; egress-dominated.
  • Core design: Resumable upload direct to blob; async chunked transcoding pipeline → multiple renditions (240p–4K) + HLS/DASH manifest; multi-tier CDN serves segments; metadata DB.
  • Data model: video(id, ownerId, status, renditions[], manifestUrl); segments in object storage.
  • Hardest part: Global low-latency delivery + the enormous storage/egress cost; transcoding throughput.
  • Key trade-off: Pre-transcode many renditions (storage/compute vs smooth playback) — lazy-transcode the long tail; CDN cost vs latency; eventual-consistent view counts.
18 · Design file storage & sync (Dropbox / Drive)
  • 🔍 Full deep-dive (with diagrams) →
  • Scale: Large files; sync across devices; offline edits; sharing.
  • Core design: Content-hashed chunks in object storage; a metadata service tracks file tree + versions; clients do delta sync (only changed chunks) and receive change notifications.
  • Data model: file(id, ownerId, path, version); chunk(hash, blobRef); device cursors.
  • Hardest part: Efficient sync of large files and resolving conflicting concurrent edits.
  • Key trade-off: Chunk dedup (bandwidth/storage savings vs metadata complexity); strong consistency vs offline editing + later merge.
19 · Design ride-hailing (Uber / Lyft)
  • 🔍 Full deep-dive (with diagrams) →
  • Scale: ~5M online drivers pinging every ~4s = ~1.25M location writes/s; matching read-heavy.
  • Core design: Location ingest → in-memory geospatial index (geohash / QuadTree / S2) sharded by region; matching service queries nearby cells, ranks by ETA; durable trip state machine; maps/ETA + surge.
  • Data model: Ephemeral driver locations (in-memory, TTL); durable trip(id, riderId, driverId, state, ts).
  • Hardest part: Indexing 1M+ moving points/s so ‘who’s near this rider’ answers in ms.
  • Key trade-off: Geohash (simple, fixed cells) vs QuadTree (adapts to density); ephemeral locations (write-speed) vs durable trips (correctness); geo-sharding vs cross-boundary matching.
20 · Design food delivery (DoorDash)
  • 🔍 Full deep-dive (with diagrams) →
  • Scale: Three-sided (customer, restaurant, courier); real-time order + courier tracking.
  • Core design: Order service + restaurant catalog/availability + courier matching (geospatial, like Uber) + payments; an event-driven order lifecycle coordinates state.
  • Data model: order(id, items, state); restaurant(menu, availability); courier locations (geo index).
  • Hardest part: Coordinating three parties in real time and accurate ETAs (prep time + travel).
  • Key trade-off: Courier assignment for speed vs efficiency (batch deliveries); availability consistency vs latency.
21 · Design ticket booking / flash sale (Ticketmaster)
  • 🔍 Read the full deep-dive (with diagrams) →
  • Scale: Limited seats; massive concurrent demand spikes; zero double-booking.
  • Core design: Seat inventory with short-lived reservations (hold + TTL); a virtual waiting room/queue throttles the spike; strongly-consistent seat lock; idempotent payment.
  • Data model: seat(eventId, seatId, state{free|held|sold}, holdExpiry, version) with optimistic locking.
  • Hardest part: Preventing double-selling the same seat under extreme contention.
  • Key trade-off: Correctness over availability for the seat lock (the one strongly-consistent spot); everything else stays highly available.
22 · Design a payment / digital wallet
  • 🔍 Read the full deep-dive (with diagrams) →
  • Scale: Money movement; never lose or double-charge; fully auditable.
  • Core design: Idempotency key on every op; a double-entry ledger as source of truth; strong consistency; async fraud checks; reconciliation jobs; sagas for multi-step distributed transactions.
  • Data model: ledger_entry(txnId, account, debit/credit, amount, ts); idempotency_keys table.
  • Hardest part: Exactly-once money movement across services that fail and retry.
  • Key trade-off: Correctness + auditability dominate — accept latency/complexity (idempotency, ledger, reconciliation) over any risk of a wrong balance.
23 · Design a distributed message queue (Kafka)
  • 🔍 Full deep-dive (with diagrams) →
  • Scale: Millions of msgs/s; multi-day retention; ordering; replay.
  • Core design: Topics split into partitions = append-only logs across brokers; producers append by key; consumer groups read by offset; replication factor + ISR + leader election.
  • Data model: partition log files (offset → record); committed consumer offsets; replica sets.
  • Hardest part: High throughput + durability while preserving ordering and handling consumer failure.
  • Key trade-off: More partitions = parallelism but no global order; at-least-once vs exactly-once (idempotent producer + txns); retention cost vs replay.
24 · Design a key-value store (DynamoDB / Cassandra)
  • 🔍 Read the full deep-dive (with diagrams) →
  • Scale: Highly available, horizontally scalable, multi-region, tunable consistency.
  • Core design: Partition via consistent hashing; replicate to N nodes; quorum reads/writes (R+W>N); gossip membership; versioning / vector clocks for conflicts; hinted handoff.
  • Data model: key → value + version vector; ring of virtual nodes.
  • Hardest part: Staying available during node/partition failures while bounding inconsistency.
  • Key trade-off: CAP: favors availability + eventual consistency; tuning R/W trades latency for consistency per call.
25 · Design a search engine
  • 🔍 Read the full deep-dive (with diagrams) →
  • Scale: Index huge corpora; low-latency ranked queries; keep fresh.
  • Core design: Crawler → indexer builds a sharded inverted index (term → postings) → query service scatters to shards, gathers, ranks (TF-IDF/BM25 + signals); cache popular queries.
  • Data model: inverted index shards; doc store; ranking signal store.
  • Hardest part: Relevance ranking + scatter-gather across a massive index within tight latency.
  • Key trade-off: Index freshness vs cost; precomputed ranking signals (fast, lag reality); more shards = parallelism + more scatter-gather overhead.
26 · Design ad-click / event analytics
  • 🔍 Full deep-dive (with diagrams) →
  • Scale: Billions of events/day; near-real-time aggregates + accurate billing.
  • Core design: Stream ingest (Kafka) → stream processor (Flink/Spark) for real-time aggregates → OLAP store (Druid/ClickHouse); a batch path reconciles for accuracy (lambda/kappa).
  • Data model: raw events (append); pre-aggregated rollups by dimension/time.
  • Hardest part: Counting accurately at huge volume despite duplicates and late/out-of-order events — it bills money.
  • Key trade-off: Real-time (fast, approximate) vs batch (accurate, delayed); exactly-once aggregation needs idempotency/dedup.
27 · Design nearby / geospatial search (Maps, Yelp)
  • 🔍 Full deep-dive (with diagrams) →
  • Scale: Find POIs within radius/region, fast, global; read-heavy.
  • Core design: Index locations with geohash / QuadTree / S2 cells so nearby points share keys; query the cell + neighbors; cache hot regions.
  • Data model: poi(id, lat, lng, cellId, attrs); cell → poi list.
  • Hardest part: Querying ‘what’s near me’ without scanning, across uneven density (city vs rural).
  • Key trade-off: Cell size: fine (precise, bigger index, more lookups) vs coarse (cheap, less precise).
28 · Design an observability / metrics system
  • 🔍 Full deep-dive (with diagrams) →
  • Scale: High-volume, high-cardinality time-series; near-real-time query + alerting.
  • Core design: Agents push metrics → ingestion tier → time-series DB (Prometheus/M3/Mimir) with downsampling + retention tiers; query layer + alerting engine; logs/traces alongside.
  • Data model: series(metric + labels) → (timestamp, value); rollups for old data.
  • Hardest part: High cardinality (many unique label sets) exploding storage + query cost.
  • Key trade-off: Retention/resolution vs cost (keep recent high-res, downsample old); cardinality limits vs granularity.
29 · Design a stock exchange / matching engine
  • 🔍 Full deep-dive (with diagrams) →
  • Scale: Ultra-low latency (µs); deterministic, fair ordering; durable; market-data fan-out.
  • Core design: In-memory order book with a single-threaded matching engine per instrument (determinism); a sequenced append-only event log for durability/replay; separate market-data publishers; co-location.
  • Data model: order book (price levels → FIFO queues); event log of sequenced orders/trades.
  • Hardest part: Microsecond latency + strict determinism/fairness while staying durable.
  • Key trade-off: Latency vs durability (in-memory speed + fast log for recovery); single-threaded match (ordering vs per-instrument parallelism).
30 · Design collaborative editing (Google Docs)
  • 🔍 Read the full deep-dive (with diagrams) →
  • Scale: Many users edit one doc simultaneously; low latency; no lost edits.
  • Core design: Clients connect over WebSocket to a document service; concurrent edits merged via Operational Transformation (OT) or CRDTs so replicas converge; periodic snapshots + op log persist state.
  • Data model: doc(id, snapshot, version); op log (docId, op, version, authorId).
  • Hardest part: Merging concurrent edits into one consistent document with no conflicts or lost changes.
  • Key trade-off: OT (compact, server-coordinated, complex) vs CRDT (no central coordination, more metadata) — both converge conflict-free.
Remember

These answers are deliberately at leadership altitude — components, bottlenecks, and trade-offs, not code. In the room, go only as deep as asked, keep narrating your reasoning, and say what would change your design.

🧠 Remember it

Thirty designs are really a few patterns retold — learn to recognize the shape.

  • Read-heavy → precompute + cache (feed, autocomplete).
  • Moving/geo → partition + spatial index (Uber, Maps).
  • Money/seats → strong consistency only where it’s needed.
