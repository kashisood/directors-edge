📖 The story

Two people typing in the same paragraph at the same instant should never clobber each other — that’s the whole challenge. Edits can’t be saved as whole documents; they must be tiny operations that merge.

The trick is making concurrent operations commute: Operational Transformation rewrites each op against the ones it raced with, while CRDTs give every character an identity so merges are conflict-free by construction. Either way, an op log plus snapshots lets anyone reconnect and converge to the exact same text.

How to think about it — step by step

Pin the hard case

Two people typing in the same spot at once must not clobber each other. Why: this single scenario drives the design.

Express edits as operations

Insert/delete ops, not whole-document saves. Why: tiny ops can be merged; full saves can't.

Make concurrent ops commute

Operational Transformation rewrites ops against each other, or CRDTs give each character an identity so merges are conflict-free. Why: this is what guarantees everyone converges.

Persist with a log + snapshots

Append ops to a log; snapshot periodically. Why: a reconnecting client can replay and catch up with no lost edits.

Name the trade-off

OT (compact, complex) vs CRDT (simpler merges, more metadata). Why: both converge; they cost differently.

1 · Requirements

Functional: multiple people edit the same document at the same time and everyone sees a consistent result, live, with no edits lost. Non-functional: low latency (keystrokes feel instant), convergence (all replicas end identical), and offline tolerance. This is a concurrency & consistency problem on shared mutable text.

2 · Capacity

The hard case isn’t many documents — it’s many editors on one document sending a stream of tiny edits that must merge without conflict.

3 · Concurrency model

  • Edits are expressed as operations (insert char at pos, delete range), not full-document saves.
  • Two families solve the merge: Operational Transformation (OT) — transform an incoming op against concurrent ops so intent is preserved; or CRDTs — data types that mathematically merge without coordination.

4 · Data model

  • doc(id, currentSnapshot, version); an append-only operation log (docId, op, baseVersion, authorId, ts).
  • Periodic snapshots compact the log so new joiners don’t replay everything.

5 · Architecture

6 · The hard part — concurrent merge

  • The problem: if A inserts at position 3 while B deletes position 1, naive application corrupts the document — positions shift under each other.
  • OT: a central server transforms each op against the ops it didn’t yet know about, preserving intent; compact but the transform logic is famously tricky.
  • CRDT: give each character a unique, ordered identity so inserts/deletes commute and merge with no central coordinator — simpler reasoning, more metadata per character.

7 · Scaling & failure

  • Shard by document; all editors of one doc share a server (or a consensus group) so ordering is simple.
  • Clients buffer ops offline and reconcile on reconnect; the op log + snapshot let a client rebuild state and resync without loss.

8 · Trade-offs

  • OT (compact, server-coordinated, complex transforms) vs CRDT (decentralized, simpler merges, more metadata).
  • Per-document single ordering point (simple) vs fully peer-to-peer (resilient, harder).

Key sub-systems

  • Document service — per-doc ordering point.
  • OT/CRDT engine — merges concurrent ops.
  • WebSocket layer — low-latency op exchange.
  • Op log — append-only history.
  • Snapshots — periodic compaction.

Bottlenecks & what to optimize next

  • Concurrent edits are the core problem — OT/CRDT make ops commute.
  • Op log grows — snapshot periodically so new joiners don't replay everything.

Common follow-up questions

How do two simultaneous edits not corrupt the doc?

Edits are operations that are transformed (OT) or designed to commute (CRDT) so all replicas converge to the same text.

OT vs CRDT?

OT is compact but needs a central coordinator and tricky transform logic; CRDTs merge without coordination but carry more metadata per character.

How does a reconnecting client catch up?

It replays ops since its last version from the op log (and a recent snapshot) to converge with no lost edits.

🧠 Remember it

Co-editing = edits as mergeable operations (OT or CRDT) that converge.

  • Represent edits as ops, not document saves.
  • OT (server transforms) or CRDT (commutative by design) merge concurrency.
  • Op log + snapshots = reconnect, replay, converge with no lost edits.
