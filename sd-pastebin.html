📖 The story

Pastebin is a URL shortener’s cousin, with one twist: the ‘value’ is a big blob of text, not a short URL. Put that blob in the wrong place (a relational DB) and you’ll pay for it at scale.

So the move is to split storage: the blob goes to cheap object storage, the database keeps only the tiny id→blob mapping and metadata, and a CDN handles popular reads.

How to think about it — step by step

Notice what's different from a link shortener

The value isn't a short URL — it's a big blob of text. Why: big blobs don't belong in a normal database.

Decide where the blob lives

Store the text in object storage (like S3); keep only a tiny row (id → blob location) in the database. Why: object storage is cheap and scales; the DB stays small and fast.

Handle reading at scale

Front popular pastes with a CDN. Why: like the shortener, reads dominate, so cache them.

Plan for expiry early

Store an expiry time and let the object store's lifecycle rules delete old data. Why: you never want to scan everything to find what to delete.

Name the trade-off

Two lookups (DB then blob) vs putting blobs in the DB (one lookup but expensive). Why: the split is worth it at any real scale.

1 · Requirements

Functional: create a text paste, get a shareable link, read it; optional expiry & visibility. Non-functional: read-heavy, cheap storage of large blobs, expiry support.

2 · Capacity

Pastes can be large (KBs–MBs), so don’t put blobs in a relational DB — store them in object storage and keep only small metadata in the DB.

3 · Architecture

4 · Key points

  • Blob vs DB: large text belongs in object storage; the DB holds the id→blob mapping + metadata.
  • Expiry: store an expireAt; a sweeper or object-store lifecycle policy removes old blobs — avoid scanning everything.
  • IDs: same short-code generation as a URL shortener.

5 · Trade-offs

  • Object store (cheap, scalable, two lookups) vs DB blobs (one lookup, expensive at size).
  • Lifecycle-policy expiry (cheap, coarse) vs precise sweeper job.

Key sub-systems

  • Write API — stores the blob and a metadata row.
  • Object storage — the paste body (S3).
  • Metadata DB — id, blob reference, TTL, visibility.
  • Read path — CDN for popular pastes.
  • Expiry sweeper — lifecycle rules delete old blobs.

Bottlenecks & what to optimize next

  • Large blobs in the DB would be the bottleneck — keep them in object storage.
  • Popular pastes hammer the origin — cache/CDN the read path.

Common follow-up questions

Why not store the text in the database?

Large blobs bloat the DB and make it slow/expensive; object storage is cheaper, scales, and offloads the read path to a CDN.

How do you expire pastes efficiently?

Store an expireAt and rely on object-store lifecycle policies (or a sweeper) so you never scan everything.

How do you support private/unlisted pastes?

Unguessable IDs plus an access check; encrypt sensitive bodies at rest.

🧠 Remember it

Pastebin = blob in object storage, metadata in the DB, CDN for reads.

  • Never store large blobs in a relational DB.
  • Object storage + lifecycle/TTL handles expiry cheaply.
  • Short-code IDs like a URL shortener.
