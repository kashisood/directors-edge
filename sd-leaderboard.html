📖 The story

A leaderboard needs two things that are individually easy and jointly hard at scale: the top N players, and your exact rank — updated in real time as scores change constantly.

A sorted set (Redis ZSET) gives both in log time. The scaling question is what happens at tens of millions of players, where exact global rank gets expensive and approximation enters.

How to think about it — step by step

Name the two queries

Top-N players, and 'what's my rank?' — both in real time. Why: the data structure must do both well.

Pick the right tool

A sorted set (Redis ZSET) keeps members ordered by score and answers both in log time. Why: it's purpose-built for ranking.

Make it durable

Snapshot the in-memory set to a database periodically. Why: a restart shouldn't wipe the standings.

Handle huge boards

At tens of millions, exact rank is costly — shard by segment or use approximate (percentile) ranks. Why: 'you're in the top 5%' is usually good enough.

Name the trade-off

Exact ranking (precise, expensive) vs approximate (cheap, fast). Why: match precision to what users actually need.

1 · Requirements

Functional: update a player’s score; get top-N; get a player’s rank and neighbors. Non-functional: real-time, scalable to millions of players.

3 · The hard parts

  • Exact rank at scale: ZRANK over tens of millions is costly; shard by segment, or use approximate ranks (percentile buckets) for huge boards.
  • Hot board: a single popular leaderboard concentrates writes — shard and aggregate, or batch updates.
  • Durability: the ZSET is in memory; snapshot periodically so a restart doesn’t lose standings.

4 · Trade-offs

  • Exact ranking (precise, costly) vs approximate (cheap, good enough for big boards).
  • Real-time updates vs batched (less load, slight lag).

Key sub-systems

  • Score API — ingests updates.
  • Sorted set — Redis ZSET for rank/top-N.
  • Sharding — by game/segment.
  • Snapshot store — durable history.
  • Read API — top-N, my-rank, neighbors.

Bottlenecks & what to optimize next

  • Exact global rank at tens of millions is the bottleneck — shard or approximate.
  • A single popular board concentrates writes — shard and aggregate or batch updates.

Common follow-up questions

How do you get a player's exact rank fast?

A sorted set answers rank in O(log n); at extreme scale, shard by segment or serve approximate percentile ranks.

How do you make it durable?

The ZSET is in memory; snapshot to a database periodically so a restart doesn't lose standings.

How do you handle cheating?

Validate score updates server-side, rate-limit, and run anomaly detection before they hit the board.
