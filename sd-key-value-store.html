Design a Key-Value Store (DynamoDB / Cassandra)

📖 The story

This is the system that taught the industry the CAP trade-off in the flesh. The goal: stay writable and fast even when nodes and networks fail — which means giving up instant global consistency.

The machinery is elegant: a consistent-hash ring spreads keys with no leader, each key is replicated N times, and tunable read/write quorums let every caller dial the consistency-versus-latency knob themselves. Conflicts are detected with version vectors and resolved deliberately.

How to think about it — step by step

Accept failure as normal

Thousands of nodes means something is always down. Why: the system must stay writable through failures.

Spread data with a hash ring

Consistent hashing places keys on a ring; no leader. Why: any node can serve, and membership changes move few keys.

Replicate with quorums

N copies; require R reads + W writes to agree, with R+W>N for strong-ish reads. Why: callers tune consistency vs latency per request.

Resolve conflicts

Version vectors detect divergence; resolve by last-writer-wins or return siblings. Why: always-writable means two writes can clash.

Name the CAP choice

It favors availability + eventual consistency. Why: stating it shows you understand the core trade-off.

1 · Requirements

Functional: get(key) / put(key, value) at massive scale, across regions. Non-functional: always-writable (high availability), horizontally scalable, low latency, and tunable consistency. This is the Dynamo/Cassandra design — the textbook study in the CAP trade-off.

2 · Capacity

Designed for petabytes across thousands of commodity nodes with no single point of failure — nodes fail constantly, so the system must treat failure as normal, not exceptional.

3 · API

  • get(key) → value(s) + version; may return siblings if there was a conflict.
  • put(key, value, context) — context carries the version vector for conflict detection.

4 · Data model

  • Opaque key → value (bytes); each value tagged with a version vector (or timestamp) for conflict resolution.
  • The key space is mapped onto a consistent-hash ring of virtual nodes.

6 · The hard parts

  • Partitioning: consistent hashing spreads keys evenly and minimizes reshuffling when nodes join/leave; virtual nodes smooth out hotspots.
  • Replication & quorum: N copies; tunable R/W quorums let each caller trade consistency for latency.
  • Conflicts: with always-writable semantics, two writes can diverge — version vectors detect it; resolve by last-writer-wins or by returning siblings to the app.
  • Failure handling: gossip spreads membership; hinted handoff lets a healthy node temporarily accept a downed node’s writes; read-repair fixes stale replicas.
📍 Real example — quorum reads/writes (R + W > N)

Replicate every key to N=3 nodes. You don’t need all 3 to answer — you need a quorum. Pick how many must acknowledge a write (W) and how many you read from (R):

N = 3 replicas

W = 2, R = 2   →  R + W = 4 > 3   →  STRONG: any read overlaps the latest write on ≥1 node
   write "x=5": node-A ok, node-B ok  (2 acks, done) — node-C catches up later
   read  "x":   ask node-B + node-C  → node-B has 5 (newest) → return 5  ✓

W = 1, R = 1   →  R + W = 2 < 3   →  FAST but stale reads possible (eventual consistency)

The diagram shows three replicas of each key. With writes needing two acknowledgements and reads consulting two replicas, the set you read always overlaps the latest write, so you get fresh data — that is R plus W greater than N. Drop to one and one and reads become faster but can return stale data.

What this teaches: if R + W > N, the set of nodes you read always overlaps the set you wrote, so you’re guaranteed to see the latest value — that’s tunable strong consistency. Lowering R or W makes reads/writes faster and keeps working when nodes are down, but you may read stale data. This single knob (used by DynamoDB and Cassandra) is how you trade consistency for availability and latency — the practical face of the CAP theorem.

7 · Scaling & failure

  • Add nodes → the ring rebalances a fraction of keys; throughput and capacity scale linearly.
  • No leader, no single point of failure — any node can coordinate; node loss just shifts ownership.
📍 Real example — WAL + LSM-tree (how writes survive a crash and stay fast)

Write-heavy stores (Cassandra, RocksDB, LevelDB) don’t update rows in place — random disk writes are slow. Instead every write is appended twice: once to a crash-safe log, once to an in-memory table that later flushes to disk in sorted order:

PUT user:7 = {plan:"pro"}
  1) append to WAL (write-ahead log) on disk     — durable instantly, sequential = fast
  2) write into MemTable (sorted map in RAM)

MemTable fills up  →  flush to disk as an immutable SSTable (sorted file)

  SSTable-3 (newest)  user:7 = {plan:"pro"}
  SSTable-2           user:7 = {plan:"free"}   ← older, shadowed
  SSTable-1           user:3 = ...

read user:7  → check MemTable, then newest SSTable first → returns {plan:"pro"}
crash before flush? → replay the WAL on restart, nothing lost.
compaction runs in background → merges SSTables, drops shadowed/deleted keys.

The diagram shows the write path. Each write is appended to a write-ahead log on disk, which makes it durable instantly, and is also added to a sorted table in memory. When that memory table fills up it is flushed to disk as an immutable sorted file. Reads check memory first, then the newest files, so the latest value wins; and if the system crashes before a flush, replaying the log recovers everything.

What this teaches: the WAL makes a write durable the instant it’s appended (sequential writes are cheap), and the LSM-tree turns random updates into fast in-memory writes plus periodic sorted flushes. Reads check newest-first, so the latest value wins; compaction later merges files and reclaims space. The trade vs a B-tree: blazing writes and great compression, but reads may touch several files (mitigated by Bloom filters per SSTable) and compaction costs background I/O.

8 · Trade-offs

  • CAP: favors Availability + Partition-tolerance with eventual consistency.
  • Tunable R/W: strong reads (R+W>N, slower) vs fast reads (small R, possibly stale).
  • Conflict resolution: last-writer-wins (simple, can lose data) vs siblings (correct, app must merge).

Key sub-systems

  • Coordinator — any node routes requests.
  • Hash ring — consistent hashing + virtual nodes.
  • Replicas — N copies, quorum R/W.
  • Anti-entropy — gossip, hinted handoff, read-repair.
  • Conflict resolution — version vectors.

Bottlenecks & what to optimize next

  • Availability under failure is the goal — quorums + hinted handoff keep writes flowing.
  • Hot partitions — virtual nodes spread load; tune R/W per workload.

Common follow-up questions

How does it stay available when nodes fail?

Quorum writes (W replicas ack), hinted handoff (a healthy node temporarily covers a downed one), and gossip-based membership.

How do you tune consistency?

Pick R and W; R+W>N gives strong-ish reads, smaller values give lower latency and higher availability.

How are write conflicts resolved?

Version vectors detect divergence; resolve via last-writer-wins (simple, can lose data) or return siblings for the app to merge.

🧠 Remember it

KV store = availability + eventual consistency on a hash ring, tunable per call.

  • Consistent hashing partitions keys; no leader, no SPOF.
  • N replicas + tunable R/W quorums (R+W>N for strong reads).
  • Version vectors detect conflicts; gossip + hinted handoff survive failure.
