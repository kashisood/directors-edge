📖 The story

Syncing files looks like ‘copy bytes’ until the files are huge, edited offline, and shared. The win is never moving more bytes than you must — only the parts that changed.

So files are split into content-hashed chunks in object storage, a metadata service tracks the tree and versions, and clients sync deltas and get change notifications. Identical chunks are stored once (dedup).

How to think about it — step by step

Aim to move fewer bytes

Only sync what changed, not whole files. Why: editing one line of a big file shouldn't re-upload it all.

Chunk and hash files

Split files into content-hashed chunks in object storage. Why: identical chunks store once (dedup), and you sync only new chunks.

Separate metadata

A metadata service tracks the file tree and versions. Why: this is the consistency-sensitive hot part; keep it apart from blobs.

Handle conflicts

Concurrent edits become versioned/conflicted copies or merge. Why: offline editing means clashes are inevitable.

Name the trade-off

Chunk dedup saves bandwidth but adds metadata complexity. Why: worth it for large files.

3 · The hard parts

  • Delta sync: hash chunks; upload only chunks the server doesn’t already have — huge bandwidth savings on small edits to big files.
  • Conflicts: concurrent edits on two devices → keep both versions (conflicted copy) or merge; versions make it recoverable.
  • Metadata at scale: the file tree + version history is the hot, consistency-sensitive part; shard by user/account.

4 · Trade-offs

  • Chunk dedup (bandwidth/storage savings) vs metadata complexity.
  • Strong consistency on metadata vs supporting offline edits + later merge.

Key sub-systems

  • Sync client — computes diffs, watches changes.
  • Metadata service — file tree + versions.
  • Chunk store — content-hashed blobs.
  • Object storage — the chunks.
  • Notification channel — pushes changes to devices.

Bottlenecks & what to optimize next

  • Re-uploading whole files is the bottleneck — sync only changed chunks (delta sync).
  • Metadata is the consistency-sensitive hot part — shard by account and keep it strongly consistent.

Common follow-up questions

How do you sync a 1GB file after a small edit?

Split files into content-hashed chunks; upload only the chunks that changed — usually a tiny fraction.

How do you handle two devices editing offline?

Versioning plus conflict resolution — keep both as a 'conflicted copy' or merge; nothing is silently lost.

Why separate metadata from blobs?

Metadata (the tree, versions) needs strong consistency and fast queries; blobs just need cheap, scalable storage.
