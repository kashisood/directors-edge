Design a URL Shortener (TinyURL)

A deep-dive into the classic read-heavy service: minting short codes without coordination, and serving billions of redirects from cache.

📖 The story

It looks trivial — map a short code to a long URL — but the interview is really about two things: generating short unique codes at scale without a bottleneck, and serving a flood of redirects fast.

So you precompute capacity (62^7 codes), hand each node a range of IDs to encode locally, and put a cache/CDN in front so the read path barely touches the database.

How to think about it — step by step

Ask: who uses this, and how?

People create a link occasionally but click it constantly. Why it matters: reads dwarf writes, so we should make reading effortless even if creating does extra work.

Do quick back-of-envelope math

Say 100M new links/day. That's only ~1,200 writes/sec, but clicks are 100× more. Why: it confirms the database is tiny but the redirect path must be fast and cheap.

Find the one genuinely hard thing

Everything is easy except making a short, unique code without all servers tripping over each other. Why: naming the real challenge stops you over-engineering the rest.

Choose a technique for the hard thing

Give each server a block of numbers to hand out (encode them in base-62), or hash the URL and retry on collision. Why: both avoid a central counter that would bottleneck.

Make the common case fly

Put a cache/CDN in front so popular links redirect without touching the database. Why: 99% of traffic is reads, so caching is the single biggest win.

State the trade-off out loud

A 301 (permanent) redirect is cacheable and fast but you lose click analytics; a 302 counts every click but hits your servers. Why: showing you see the cost is what senior sounds like.

1 · Requirements

Functional: shorten a long URL; redirect a short code to it; optional custom alias, expiry, analytics. Non-functional: extremely read-heavy, very low redirect latency, high availability.

2 · Capacity

~100M new links/day → ~1.2K writes/s; reads 100× that (tens of thousands/s). 7-char base62 = 62^7 ≈ 3.5 trillion codes — plenty. Years of links × ~500 bytes → tens of TB — trivial for a KV store.

3 · Architecture

4 · The hard part — code generation

  • Counter range: each app node grabs a block of IDs (e.g. 1,000 at a time) from a central allocator and base62-encodes them — no per-request coordination, no collisions.
  • Hashing alternative: hash(longURL), take 7 chars, retry on collision — no coordination but must detect collisions.
  • Custom aliases: a uniqueness check on the chosen alias.

5 · Trade-offs

  • Counter (compact, sequential, slightly guessable) vs hash (no coordination, collision handling).
  • 301 permanent redirect (cacheable, fast, loses per-hit analytics) vs 302 (every hit counted, more origin load).

Key sub-systems

  • Write API — validates the URL, mints a code, persists it.
  • Code generator — base62 of an ID range, or hash + collision check.
  • KV store — code → longURL, the source of truth.
  • Redirect path — cache/CDN in front of a stateless app tier.
  • Analytics — click events written asynchronously.

Bottlenecks & what to optimize next

  • The redirect read path is hottest — scale it first with cache/CDN and read replicas.
  • Central code generation can bottleneck — hand out ID ranges so nodes never coordinate per request.

Common follow-up questions

How do you stop two servers minting the same code?

Give each server a pre-allocated range of IDs (or hash with a collision check) so there's no shared counter on the hot path.

Why pick 302 over 301?

302 isn't browser-cached, so every click reaches you and can be counted — at the cost of more load. 301 is faster but loses analytics.

How do you handle malicious links?

Scan URLs on creation against threat feeds, rate-limit creation per account, and support blocklists/takedowns.

🧠 Remember it

URL shortener = cheap reads via cache + coordination-free code generation.

  • Cache/CDN absorbs the read flood; KV store is the source of truth.
