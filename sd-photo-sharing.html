📖 The story

Photo sharing is a feed plus a media problem. The feed reuses the fan-out ideas from the News Feed; the new challenge is storing and serving images globally, fast, while absorbing upload bursts.

So images go to object storage and a CDN, an async pipeline makes the many sizes you need, and metadata lives in a sharded DB — keeping the read path cheap.

How to think about it — step by step

Split it into two problems

A feed (like Twitter) plus media handling. Why: reuse the feed solution and focus energy on images.

Get images out of the database

Store photos in object storage and serve via CDN; keep only metadata in the DB. Why: media is huge and read globally; the DB shouldn't touch it.

Process uploads asynchronously

On upload, accept it, then generate thumbnails/sizes in the background. Why: the user shouldn't wait, and you need many sizes for different screens.

Reuse the feed pattern

Fan-out timelines with the celebrity hybrid. Why: don't reinvent it — connect to what you already know.

Name the trade-off

Pre-making sizes costs storage but makes serving instant. Why: worth it for popular content.

1 · Requirements

Functional: upload photos, follow users, view a home feed, like/comment. Non-functional: read-heavy image serving, low latency globally, bursty uploads.

2 · Capacity

Reads (views) vastly outnumber writes (posts). Media dominates storage — petabytes — so it lives in object storage + CDN, never the DB.

3 · Architecture

4 · The hard parts

  • Media delivery: pre-generate sizes and serve via CDN for low latency; originals stay in cheap storage.
  • Feed: same fan-out-on-write + celebrity hybrid as the News Feed deep-dive.
  • Upload bursts: accept the upload, process (thumbnails) asynchronously, mark ready when done.

5 · Trade-offs

  • Pre-generate sizes (storage cost, fast serve) vs on-the-fly resize (compute, latency).
  • Fan-out write vs read — hybrid for high-follower accounts.

Key sub-systems

  • Upload service — accepts originals to object storage.
  • Processing pipeline — async thumbnails/resolutions.
  • Metadata DB — sharded posts/likes/comments.
  • Feed — fan-out timeline like the News Feed.
  • CDN — serves all images.

Bottlenecks & what to optimize next

  • Image egress dominates — maximize CDN cache-hit ratio and pre-size images.
  • Upload spikes — accept fast, process thumbnails asynchronously.

Common follow-up questions

Where do images live?

Originals in object storage; pre-generated sizes served via CDN. The DB holds only metadata.

How do you handle upload bursts?

Accept the upload immediately, then generate thumbnails/sizes in a background pipeline and mark the post ready when done.

How is the feed built?

Same fan-out-on-write timeline cache as Twitter, with pull-on-read for very high-follower accounts.
