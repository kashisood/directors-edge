📖 The story

‘Find restaurants within 2 km’ sounds easy until you have millions of places and can’t scan them all. You need an index where geography itself is the key — nearby points stored together.

That’s geohashing (or QuadTree/S2): encode location so a shared prefix means physical closeness, then query the relevant cell plus its neighbors. Cell size trades precision against index size.

How to think about it — step by step

State the impossible-by-brute-force part

You can't scan millions of places per query. Why: it forces a spatial index.

Make geography the key

Geohash/QuadTree/S2 give nearby places the same cell key. Why: 'near me' becomes 'look up my cell'.

Query the neighborhood

Check the user's cell plus its neighbors. Why: a place just over a cell border must still be found.

Adapt to density

QuadTree/S2 subdivide where it's crowded (cities). Why: fixed cells are too coarse downtown and too fine in the countryside.

Name the trade-off

Fine cells = precise but bigger index; coarse = cheap but rough. Why: pick based on your density.

1 · Requirements

Functional: given a location, return points of interest within a radius/region, ranked. Non-functional: low latency, global scale, read-heavy, handle uneven density (cities vs countryside).

2 · Architecture

📍 Real example — a geohash

Take San Francisco — latitude 37.7749, longitude -122.4194. Encode it as a geohash and you get a short string where each extra character zooms into a smaller box:

37.7749, -122.4194  →  geohash  9q8yyk8ytpxr

9q          →  ~1250 km box  (most of N. California / Nevada)
9q8yy       →  ~2.4 km box   (central San Francisco)
9q8yyk8     →  ~150 m box    (a couple of city blocks)
9q8yyk8ytpxr→  ~1 m box      (basically a point)

What this teaches: nearby places share a geohash prefix, so “find things near me” becomes “find rows whose geohash starts with 9q8yy” — a cheap index lookup instead of scanning every point. One catch: two close points can sit just across a cell boundary with different prefixes, so you always query the matching cell plus its 8 neighbours, then filter by exact distance. (Values are approximate — geohash precision is well defined per length.)

3 · The hard parts

  • Indexing: geohash gives fixed-size cells (simple); QuadTree/S2 subdivide where density is high (adapts to cities).
  • Boundaries: a point near a cell edge needs neighbor cells too — always query the ring of neighbors.
  • Ranking: distance plus business signals (rating, popularity).

4 · Trade-offs

  • Fine cells (precise, bigger index, more lookups) vs coarse (cheap, less precise).
  • Geohash (simple) vs QuadTree/S2 (density-adaptive, complex).

Key sub-systems

  • Geo index — geohash/QuadTree/S2 cells.
  • Query service — cell + neighbor lookup.
  • POI store — attributes/metadata.
  • Ranking — distance + business signals.
  • Region cache — hot areas.

Bottlenecks & what to optimize next

  • Scanning all points is impossible — the spatial index makes 'near me' a cell lookup.
  • Uneven density — QuadTree/S2 subdivide where it's crowded.

Common follow-up questions

How do you query 'within 2km' without scanning everything?

Index by geohash/QuadTree so nearby places share a cell; query the user's cell plus neighbors and filter by exact distance.

How do you handle dense cities vs sparse areas?

QuadTree/S2 subdivide cells where density is high, so each cell holds a manageable number of points.

Straight-line vs road distance?

Cells give candidates by straight-line proximity; a routing service computes true travel distance/time for ranking.

🧠 Remember it

Geo search = geohash/QuadTree index; query cell + neighbors.

  • Nearby points share a cell key — no full scan.
  • Always include neighbor cells for edge cases.
  • Cell size = precision vs cost; S2/QuadTree adapt to density.
